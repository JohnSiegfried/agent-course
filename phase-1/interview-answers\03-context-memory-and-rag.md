# Q41-Q48：Context、Memory 与压缩标准答案

## Q41. 本地上下文和模型可见上下文有何区别？

- **来源映射**：`INT-NC-B02/B04` 报告上下文大小、压缩和记忆问题。
- **30 秒答案**：本地上下文是 Runtime 可访问的全部状态、文件、日志和记忆；模型可见上下文是经过检索、排序、裁剪、脱敏和权限过滤后放进本次请求的有限投影。
- **2 分钟标准答案**：不能把“进程知道”偷换成“模型知道”。Runtime 可以保存完整任务状态、artifact、工具结果、审计日志和密钥引用，但每次模型调用只选择完成当前步骤所需的信息。选择过程要考虑相关性、时效、可信度、权限、Token 成本和位置；敏感值不进 Prompt，原始大文件只提供受控摘要或引用。模型输出也不能直接改本地真实状态，必须通过 typed decision 和 Runtime 校验。
- **深挖**：为什么不把全部历史都发给模型？成本、注意力稀释、冲突、泄露和注入；本地可见是否等于当前用户可见？不是，还需 actor/tenant 过滤；模型如何请求更多上下文？通过受控检索 Tool。
- **项目映射**：ESGQA `P1-ESG-MEMORY-001` 组装 `memory_context`；agentHub `P1-HUB-CONTEXT-001` 按权重和预算生成 digest。
- **评分/危险**：“上下文就是聊天记录”忽略状态、权限和选择层。

## Q42. 写出上下文预算方程。

- **来源映射**：B 级面经中的上下文容量、压缩和输出窗口追问。
- **30 秒答案**：`W >= S + H + T + R + O + M`：总窗口至少容纳系统指令、历史/状态、Tool Schema、检索内容、输出预留和安全余量。
- **2 分钟标准答案**：更工程化地写：`available_input = model_window - reserved_output - protocol_overhead - safety_margin`，输入端再分配给 system/policy、working state、conversation、tool definitions、retrieved evidence 和当前请求。预算不是平均分配：不可丢的安全策略和当前目标先保留，证据按价值/Token 排序，历史可摘要。Tool 调用后还要为 Observation 和后续生成留空间，不能在第一轮用满窗口。
- **深挖**：为什么要 safety margin？Tokenizer 估算、Provider 包装和动态 ToolResult 有误差；输出预留多少？按任务类型、Schema 和历史分布；字符数能代替 Token 吗？只能粗估，最好用目标 tokenizer。
- **项目映射**：agentHub `getProjectDigest(maxTokens)` 有预算思想，但源码中的字符截断是启发式，不能说严格满足 Token 方程。
- **评分/危险**：只回答“窗口 128K”没有说明输入输出共同占用。

## Q43. 上下文超限时先删什么？

- **来源映射**：`INT-NC-B02/B04` 的过长上下文处理；Q95 会扩展阈值选择。
- **30 秒答案**：不按时间简单从旧到新删除；先去重和移除低价值噪声，再外置大 artifact，摘要已验证历史，保留当前目标、安全策略、未完成状态、关键证据和最近 ToolResult。
- **2 分钟标准答案**：先做分类：必须保留的是目标、约束、actor/policy、当前计划、未决审批、关键事实和来源；可压缩的是已完成步骤、重复对话和旧工具输出；可外置的是代码、大文档和原始日志；可删除的是无关寒暄、重复错误和过期候选。压缩后要保留 provenance、open questions、decisions 和不可逆动作状态，并用回放集验证任务成功率，而不是只验证摘要通顺。
- **深挖**：最老消息一定最不重要吗？不是，可能是用户约束；摘要能否继续摘要？可以但会累积损失，应定期从原始事实重建；Tool Schema 太大怎么办？只暴露当前最小工具集。
- **项目映射**：`P1-HUB-COMPRESSION-001` 有 summary + pending prompt 恢复；没有摘要保真评分证据。
- **评分/危险**：机械保留最近 N 轮会丢掉早期关键约束。

## Q44. 长窗口为何仍需要检索与排序？

- **来源映射**：`INT-NC-B04` 的长上下文限制；A/B 面经中的 RAG 选型。
- **30 秒答案**：窗口变长只扩大容量，不保证模型能定位正确证据；检索与排序还能降低成本、延迟、冲突、敏感暴露和无关信息干扰。
- **2 分钟标准答案**：长窗口解决“能不能放进去”，不等于“放进去后能可靠使用”。真实知识库还需要权限过滤、版本、时效和 provenance；把全部文档塞入 Prompt 会造成位置效应、重复/矛盾证据、Token 成本和攻击面扩大。检索先缩小候选，排序和 rerank 提升证据位置，Context builder 再做多样性、去重和预算。对于小而稳定的文档，长窗口可以简化 Pipeline，但仍需版本和引用。
- **深挖**：什么时候不用向量检索？数据小、结构强、可直接 SQL/规则过滤；检索一定提升正确率？不一定，错误召回会误导；长窗口与 RAG 如何评测？固定生成器比较 evidence recall、faithfulness、成本和延迟。
- **项目映射**：ESGQA `P1-ESG-RETRIEVAL-001` 体现混合检索；没有当前指标证明优于全量长上下文。
- **评分/危险**：“窗口够长就不需要 RAG”忽略定位和治理。

## Q45. 记忆系统的生命周期是什么？

- **来源映射**：`INT-DY-C01`、`INT-NC-B02` 反复出现 Memory；Q102 进一步讲短转长。
- **30 秒答案**：capture -> validate/classify -> store -> retrieve/rank -> use -> update/merge -> expire/delete -> audit；写入和读取都要策略，不是把聊天永久向量化。
- **2 分钟标准答案**：候选记忆先判断是否值得跨任务保存、是否含敏感信息、来源是否可信，再结构化为事实、偏好、决策或未解决问题，并记录 user/tenant、时间、source、confidence 和 TTL。读取时按 actor 权限、相关性、时效和冲突选择；使用后通过反馈更新引用和可信度。新旧记忆冲突不能简单覆盖，需版本、合并或询问用户。必须支持查看、更正、遗忘和审计，避免错误摘要和 Prompt Injection 长期污染。
- **深挖**：向量库就是长期记忆吗？只是索引/存储手段；所有对话都写吗？不，写入需门槛；如何删除？删除原文、索引、副本和缓存，并记录合规审计。
- **项目映射**：ESGQA 有近期对话、长期画像和 FAISS 兜底，但缺 TTL、用户编辑和敏感治理，证据 `P1-ESG-MEMORY-001`。
- **评分/危险**：只讲“短期放 Redis、长期放向量库”没有语义生命周期。

## Q46. 压缩为什么要状态机和迟滞？

- **来源映射**：`INT-NC-A01/B04` 的多级压缩和上下文压缩追问。
- **30 秒答案**：压缩会与新请求、工具事件和 Provider 生命周期竞态；状态机保证触发、摘要、切换和恢复顺序，迟滞避免阈值附近反复压缩。
- **2 分钟标准答案**：至少有 normal、compression_requested、compressing、restarting/restoring、failed 状态。触发后冻结或排队新 Prompt，等待在途动作达到安全点，生成并验证摘要，保存 checkpoint，再切换到新上下文。迟滞可用高水位触发、低水位恢复，例如 75% 触发，压到 45% 以下才回 normal；还要冷却时间和最小增量，防 usage 在阈值附近抖动。失败时恢复原会话或安全终止，不能丢 pending prompt。
- **深挖**：为什么单个 if 不够？压缩期间可能到达事件；摘要为空怎么办？fallback 并标记质量风险；阈值如何定？见 Q95，以任务质量、成本和延迟实验确定。
- **项目映射**：agentHub 有 70% 阈值和 pending prompt 状态流，`P1-HUB-COMPRESSION-001`；没有迟滞和保真评分证据。
- **评分/危险**：每次超过阈值立即重启 Provider，可能造成压缩风暴和请求丢失。

## Q47. ESGQA 上下文实现如何评价？

- **来源性质**：项目源码评审题。
- **30 秒答案**：优点是意图识别前组合近期对话、长期画像和 FAISS 兜底；缺点是固定用户、无 Token 预算/TTL/编辑治理，且数据库与向量异常处理不一致。
- **2 分钟标准答案**：`node_memory()` 从 SQLite 取最近 4 条和长期画像，再追加 FAISS 相似搜索结果，说明项目意识到时间近因、用户画像和语义召回三类来源。边界是 `user_id` 固定为 `user_001`，没有租户/用户隔离证据；拼接没有 Token 预算、冲突消解、来源权重和敏感过滤；SQLite 异常降级为空，但 FAISS 调用没有在同一函数内统一捕获。改进优先级是身份边界、typed memory、预算排序、写入/删除策略和评测，而不是先增加更多记忆来源。
- **深挖**：最近 4 条依据是什么？源码常量事实，不是已验证最优；长期画像如何更新？当前证据不足；如何测污染？构造错误旧偏好、越权用户和恶意记忆回放。
- **项目映射**：`P1-ESG-MEMORY-001`。
- **评分/危险**：不能声称“完整企业级长期记忆”或杜撰用户画像准确率。

## Q48. agentHub ContextBus 与压缩有什么亮点和缺口？

- **来源性质**：项目源码评审题；对应多条真实上下文面经。
- **30 秒答案**：亮点是 ContextEntry 结构化、版本化、按类型/时间/引用加权，并在预算内生成 digest；压缩期间显式保存 pending prompt。缺口是启发式权重、字符截断、硬 70% 阈值、无迟滞和摘要保真评测。
- **2 分钟标准答案**：ContextBus 把 convention、decision、known-issue 等作为有作者、状态、标签、版本的条目，查询后按基础权重、7 天衰减和引用奖励排序，并支持归档/GC，比整段聊天更可治理。Runtime 根据 Usage 超过 70% 设置压缩，在下一请求到来时暂存 prompt、请求 summary、重启 Provider，再用 summary + pending prompt 恢复。风险是排序与截断是启发式，摘要可能丢不变量；压缩/新事件仍需竞态测试；也没有证据证明跨 Provider 恢复语义一致。
- **深挖**：为什么 ContextBus 不等于 Memory 全生命周期？写入质量、用户纠错和删除治理不完整；字符截断有什么问题？可能切断 JSON/代码/语义；如何改进？tokenizer、typed summary、保真回放、迟滞和版本化 checkpoint。
- **项目映射**：`P1-HUB-CONTEXT-001`、`P1-HUB-COMPRESSION-001`。
- **评分/危险**：高分必须同时指出实现亮点和未验证假设。
