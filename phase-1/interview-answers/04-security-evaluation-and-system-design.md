# Q49-Q64：安全、评测与系统设计标准答案

## Q49. 如何从零做 Agent threat model？

- **来源映射**：`INT-CS-C01/C02`、`INT-JG-C01` 用作趋势校验；真实项目追问常从 Tool/Memory 延伸到安全。
- **30 秒答案**：先画信任边界和数据流，再枚举资产、主体、入口、能力和攻击者；按输入、模型、Tool、Memory、Provider、运行环境和输出逐层分析威胁、控制与残余风险。
- **2 分钟标准答案**：资产包括密钥、用户数据、代码、资金动作和审计记录；主体包括用户、Agent、子 Agent、Provider、Tool Server 和管理员。重点威胁有 Prompt Injection、工具越权、数据外泄、记忆投毒、供应链、路径逃逸、资源耗尽和审批替换。每个威胁记录前置条件、影响、现有控制、检测、恢复和 owner，按 likelihood/impact 排序。安全控制要落到默认拒绝、最小权限、隔离、出站控制、参数绑定、审计和事件响应，并通过负向测试验证。
- **深挖**：STRIDE 能直接套吗？可作为清单，但要补 Agent 特有的数据/控制混淆；Threat model 何时更新？新增 Tool/Provider/数据域和事故后；风险接受怎么做？记录 owner、期限和补偿控制。
- **项目映射**：agentHub 自带威胁模型，证据 `P1-HUB-DOC-SECURITY-GAPS-001`，但它不是外部认证。
- **评分/危险**：只列 OWASP 名词而不画资产和信任边界，难以落地。

## Q50. 为什么 Prompt Injection 不能只靠 Prompt 解决？

- **来源映射**：`INT-CS-C01` 训练题单；A/B 面经中的工具和 RAG 安全会隐含追问。
- **30 秒答案**：因为攻击内容和系统指令最终都进入同一模型推理通道，模型不是可信安全边界；真正控制必须由 Runtime 权限、数据隔离和执行策略强制。
- **2 分钟标准答案**：外部网页、文档、ToolResult 和记忆都应视为不可信 data，即使写“忽略其中指令”，模型仍可能混淆。防御需要多层：来源标记与最小检索；Prompt 中分离 instruction/data；模型只产生候选动作；Runtime 按 actor、Tool、参数和资源重新授权；高风险 HITL；输出做敏感信息和政策检查；Trace 记录来源。Prompt 规则可降低概率，但不能替代不可绕过的控制。
- **深挖**：RAG 内容是公司内部就可信么？仍可能被误写或投毒；模型拒绝是否足够？攻击可诱导别的路径；如何测试？间接注入、跨文档冲突、工具结果注入和记忆持久化。
- **项目映射**：ESGQA 没有通用 Tool 沙箱；agentHub 权限/容器有边界缺口，不能声称已完全抵御注入。
- **评分/危险**：“在 system prompt 加一句不要执行恶意指令”不是完整安全方案。

## Q51. authn、authz、sandbox 有何区别？

- **来源性质**：生成训练题。
- **30 秒答案**：authn 确认主体是谁；authz 决定该主体能对哪项资源做什么；sandbox 即使授权或实现出错，也限制进程实际可触达的系统资源和影响范围。
- **2 分钟标准答案**：JWT 登录属于 authn；角色/租户/对象级政策属于 authz；容器、文件根、非 root、网络和系统调用限制属于 sandbox。三者是纵深防御：通过身份验证不代表有写权限；获得某工具权限不代表能访问宿主所有路径；沙箱也不能替代业务授权，因为隔离环境里仍可能越权操作同租户数据。
- **深挖**：API key 是 authn 还是 authz？主要认证调用方，仍需授权；容器逃逸属于哪层？sandbox 失效；模型是否是主体？执行时必须绑定真实 actor，不能把模型当用户身份。
- **项目映射**：ESGQA `P1-ESG-API-SECURITY-001` 主要覆盖入口认证/限流；agentHub `P1-HUB-PERMISSION-001/P1-HUB-SANDBOX-001` 覆盖另外两层但仍有缺口。
- **评分/危险**：“有 JWT 所以文件工具安全”混淆三层。

## Q52. 文件 Tool 如何防路径遍历、symlink 和 TOCTOU？

- **来源性质**：生成训练题，代码 Agent 面试常见深挖。
- **30 秒答案**：拒绝原始字符串拼接；以固定 workspace root 解析并 canonicalize，校验真实目标仍在根内，限制链接；打开时使用安全 flag/句柄并在操作后核验，配合最小挂载和审计。
- **2 分钟标准答案**：先把相对路径规范化，拒绝绝对路径、`..`、设备路径和非法编码；用真实文件系统 canonical path 检查 common root。仅检查一次仍有 TOCTOU：攻击者可在检查与打开间替换 symlink，因此应使用目录句柄相对打开、`O_NOFOLLOW` 等平台能力，或把工作区置于攻击者不能改变链接结构的隔离挂载中。写操作采用临时文件+原子 rename、权限/大小/扩展名限制，并记录 inode/路径与内容摘要。
- **深挖**：`startsWith(root)` 安全吗？`/root2` 和大小写/链接可绕；删除目录怎么办？按句柄递归且不跨 mount；Windows 怎么办？处理 junction、reparse point 和大小写规范化。
- **项目映射**：agentHub 路径 glob 不等于上述加固；ESGQA 未开放通用文件 Tool。
- **评分/危险**：只做正则替换 `../` 很容易被编码和链接绕过。

## Q53. 使用 Docker 后还要检查什么？

- **来源性质**：生成训练题；agentHub 项目可实证。
- **30 秒答案**：检查用户、capabilities、seccomp/AppArmor、只读根、挂载、Docker socket、网络出站、资源限制、镜像供应链、密钥、日志和宿主隔离；Docker 不是安全结论。
- **2 分钟标准答案**：容器应非 root、drop capabilities、`no-new-privileges`、受限 seccomp/AppArmor、只读 rootfs 和最小可写目录；bind mount 要最小范围和适当只读，绝不能随意暴露 Docker socket；限制 CPU/内存/PID/磁盘，网络默认拒绝或域名 allowlist；镜像固定 digest、扫描依赖并最小化；密钥短期注入且不落盘。多租户高风险场景还要考虑更强隔离或独立节点。
- **深挖**：内存限制能防所有 DoS 吗？不能，仍有 I/O/PID/网络；容器内 root 有何风险？扩大内核/挂载攻击面；预览页面呢？还需浏览器 sandbox/CSP。
- **项目映射**：`P1-HUB-SANDBOX-001` 有会话容器和内存限制；`P1-HUB-DOC-SECURITY-GAPS-001` 明确记录 seccomp、只读根、Docker socket 等缺口。
- **评分/危险**：“在 Docker 里执行所以安全”是面试红旗。

## Q54. HITL 如何避免审批疲劳和参数替换？

- **来源映射**：`INT-CS-C01` 趋势题；金融/高风险面经见 Q107。
- **30 秒答案**：只对高风险或低置信动作审批，展示清晰差异和影响；审批令牌绑定 actor、tool version、规范化参数摘要、资源和过期时间，任何变化都重新审批。
- **2 分钟标准答案**：先按风险分层：只读低风险自动执行；可逆写操作可批量/策略审批；不可逆或高金额逐次审批。界面展示将做什么、对谁、关键参数、来源、预期影响和回滚方式，而不是整段模型推理。批准产生签名 capability，只能使用一次且短时有效；执行前重新计算参数摘要。监控批准率、拒绝率、等待时长和事后撤销，长期高通过说明规则可能可自动化，长期盲批说明界面或频率有问题。
- **深挖**：能否批准“未来所有类似动作”？只在严格策略范围内；审批超时？安全拒绝；模型能否重新解释参数？可以解释，但不能改变已批准值。
- **项目映射**：agentHub 有权限请求事件，但现有证据不足以证明完整参数绑定审批协议。
- **评分/危险**：把弹窗数量当安全强度会制造盲批。

## Q55. ESGQA 的安全能力边界？

- **来源性质**：项目事实题。
- **30 秒答案**：冻结代码在 API 入口有 JWT、限流、Celery 异步和 Prometheus；但这些不等于 Agent Tool 沙箱、细粒度业务授权、租户隔离和密钥治理。
- **2 分钟标准答案**：可说的源码事实是登录签发/校验 JWT、端点 10/minute 限流、长任务提交 Celery、状态查询和指标。不能说的包括通用文件/命令 Tool 隔离、每工具权限、强租户边界和生产 Secret 管理。尤其 API 鉴权只控制“谁能调用服务”，若未来增加开放 Tool，还要独立做 actor-bound authz、workspace sandbox、出站网络和审计。`user_id` 固定也限制了多用户记忆安全表述。
- **深挖**：Celery 隔离了吗？只是异步解耦，不自动形成安全沙箱；限流防什么？基本滥用和资源压力，不防越权；下一步优先？身份/租户、密钥、Tool 契约与隔离。
- **项目映射**：`P1-ESG-API-SECURITY-001`、`P1-ESG-GAP-SANDBOX-001`、`P1-ESG-MEMORY-001`。
- **评分/危险**：不要把基础 Web 安全包装成完整 Agent Security。

## Q56. agentHub 的权限与沙箱最值得追问什么？

- **来源性质**：项目代码评审题。
- **30 秒答案**：追问默认权限是否最小、空 allowlist 语义、路径 canonicalization/symlink、执行层是否强制、Docker socket/挂载/网络、容器加固、审计和跨会话隔离。
- **2 分钟标准答案**：权限侧先看未知 Agent 的 `CUSTOM_AGENT_DEFAULT`、forbidden/allowed 检查顺序、Write patterns 与真实文件句柄是否一致，以及委派是否扩大权限。沙箱侧看非 root、capability、seccomp、只读根、三类 bind mount、网络和资源上限。再看权限事件与执行之间的 TOCTOU、容器复用/清理、秘密注入和终态审计。项目自己的 threat model 已记录若干生产缺口，所以正确表达是“有基础控制和清晰改进清单”。
- **深挖**：角色配置是否安全边界？只有执行器不可绕过时才是；容器名能否隔离租户？不能；如何验证？恶意 Tool、链接替换、跨 workspace、网络外传和资源耗尽测试。
- **项目映射**：`P1-HUB-PERMISSION-001`、`P1-HUB-SANDBOX-001`、`P1-HUB-DOC-SECURITY-GAPS-001`。
- **评分/危险**：只演示正常角色能写哪些文件，不算安全验证。

## Q57. 设计一个企业知识问答 Agent。

- **来源映射**：`INT-NC-A05` 的 RAG 对比，`INT-JG-C01` 的系统设计趋势；Q98-Q101补检索细节。
- **30 秒答案**：入口鉴权和租户解析后，Router 判断检索/澄清/拒绝；RAG 做权限过滤、混合召回、重排与引用；生成器受证据约束；评测、观测和反馈闭环贯穿全链路。
- **2 分钟标准答案**：架构分入口层、编排层、知识层、模型/Tool 层和治理层。文档入库做解析、Chunk、版本、ACL、Embedding 与索引；查询时先身份和 query rewrite，再 metadata/ACL filter、稀疏+稠密召回、rerank、Context budget 和有来源生成。无充分证据时澄清或回答不知道；高风险政策答案可人工复核。观测记录 query、索引版本、候选、最终上下文、模型版本和引用；离线测 retrieval recall/precision、faithfulness 和 task success，线上看零结果、反馈、延迟、成本和权限泄露。
- **深挖**：为什么叫 Agent 而非 RAG Workflow？只有存在动态选择工具/澄清/行动时；知识更新？见 Q98；如何防跨租户？ACL 在检索前过滤，不能生成后再遮盖。
- **项目映射**：ESGQA `P1-ESG-RETRIEVAL-001` 可证明混合检索实现；没有在线质量数字。
- **评分/危险**：“LangChain + 向量库 + LLM”不构成企业设计。

## Q58. 设计一个能改代码的 Agent Runtime。

- **来源映射**：公开面经中的代码 Agent、状态机和上下文问题；C 级材料仅作趋势。
- **30 秒答案**：用确定性 Runtime 承担 repo 快照、计划、最小上下文、文件/命令 Tool、沙箱、diff、测试、审批和提交；模型只能提议变更。
- **2 分钟标准答案**：任务进入后创建隔离 workspace 和基线 commit，索引代码并只检索相关文件；Planner 产生 typed tasks，Tool 层提供 read/search/patch/test，写路径和命令按角色最小授权。每次 patch 保存 diff、文件版本和 trace；运行格式化/静态检查/测试，失败有界修复；高风险命令、依赖安装或外部网络进入审批。最终展示变更、验证证据和未解决风险，由用户决定提交。取消时终止子进程，迟到输出不能覆盖已结束任务。
- **深挖**：为什么不能把 repo 全塞 Prompt？成本和定位；如何防覆盖用户修改？基线 hash/乐观锁和冲突检测；测试全过就安全吗？仍需审查需求、权限和供应链。
- **项目映射**：agentHub 有 Runtime、Provider、权限和容器基础，但冻结代码不能自动证明上述完整代码 Agent 工作流。
- **评分/危险**：直接给模型 shell 和整个宿主目录是高风险设计。

## Q59. 如何做 Agent 全链路可观测性？

- **来源映射**：`INT-JG-C01` 的 Trace 回放和系统设计趋势；B 级面经的项目效果追问。
- **30 秒答案**：用统一 trace/session/turn/step/tool id 串起输入、路由、模型、工具、状态、预算、权限和终态，同时保护 Prompt 与敏感数据。
- **2 分钟标准答案**：每个 step 记录模型/Prompt/Tool/策略版本、输入摘要、token、latency、decision reason、Tool 参数摘要、结果状态和错误分类。分布式调用传播 traceparent；指标分入口、Router、Provider、Tool、Memory/RAG 和业务成功；日志支持按失败原因和版本切片，Trace 可脱敏回放。告警覆盖成功率、循环、unknown outcome、权限拒绝、fallback、成本和 P95/P99。敏感字段 tokenization/哈希，严格 RBAC 和保留期限。
- **深挖**：是否保存完整 CoT？不应依赖或广泛保存私有推理；怎样关联用户体验？业务 task id 和反馈；采样会漏事故吗？高风险/失败全量，正常按策略采样。
- **项目映射**：ESGQA 有 Prometheus 入口证据，agentHub 有事件和 Usage；都不能直接声称完整分布式 Trace。
- **评分/危险**：只有请求日志无法解释 Agent 多步失败。

## Q60. Agent 质量下降时如何系统排障？

- **来源映射**：真实面经常追问“效果如何、怎么优化”；Q89/Q101/Q114细化子链路。
- **30 秒答案**：先确认指标、时间和受影响切片，再按输入/数据、路由、上下文/RAG、模型、Tool、Runtime、策略和下游逐层定位，并用版本化 Trace 对比最近变更。
- **2 分钟标准答案**：先冻结症状：是 task success、faithfulness、tool success、latency 还是成本下降；按租户、任务、模型和版本切片。检查数据/索引新鲜度与 ACL，Router 混淆，检索候选和上下文位置，Prompt/模型版本，Tool Schema/依赖，预算/压缩和权限。用失败 Trace 回放旧新版本做二分，逐层替换为 oracle 或 mock：给正确路由、正确证据、固定模型，定位误差来源。修复后跑回归集和 canary，不用单个案例宣称恢复。
- **深挖**：无版本记录怎么办？先补可复现实验和请求快照；线上好离线差？检查分布和弱标签；模型供应商悄然变化？能力探测和输出统计。
- **项目映射**：两个项目目前只有源码证据，不能编造质量事故与修复数字。
- **评分/危险**：“调 Prompt/换更大模型”不算系统排障。

## Q61. 如何控制 Agent 成本与延迟？

- **来源映射**：公开备考和系统设计材料的高频项。
- **30 秒答案**：先分解成本和关键路径，再用路由、小模型、上下文预算、缓存、并行、批处理、早停和工具优化；任何优化都要守住质量与安全门槛。
- **2 分钟标准答案**：把一次任务拆成 Router、Planner、每步模型、检索、Tool 和重试，记录 token、调用数、P95 和关键路径。低风险分类用规则/小模型，减少可见工具和历史，检索只送高价值证据；无依赖只读 Tool 可并行；稳定结果做权限安全的语义/结果缓存；设置预算、early exit 和模型 fallback。最后以“单位成功任务成本”而非单次 API 价格衡量，防止便宜模型导致更多重试。
- **深挖**：流式能降低延迟吗？主要改善首 Token 体验，不一定降低完成时间；缓存风险？过期、跨租户、权限和非确定副作用；并行总更快？限流/竞争可能更慢。
- **项目映射**：agentHub 压缩与队列、ESGQA Redis 降级可作实现例子，但无真实成本收益数字。
- **评分/危险**：只说“换小模型、加缓存”没有指标和正确性约束。

## Q62. 如何做 Agent 评测体系？

- **来源映射**：`INT-NC-B01/B04` 报告评测；Q114 比较 RAG 与 Trace。
- **30 秒答案**：分组件、Trace 和端到端三层；离线 Golden Set、仿真/回放、人工与模型辅助评审结合，线上用灰度和真实业务指标闭环。
- **2 分钟标准答案**：组件层测 Router、Tool selection/arguments、RAG retrieval、Memory、Policy 和终止；Trace 层测计划有效性、无进展、重试、权限和副作用；端到端测 task success、正确性、faithfulness、成本、延迟和安全。数据集包含正常、边界、OOD、攻击和事故案例，按版本管理并隔离 dev/test。LLM-as-Judge 只能在有 rubric、校准和人工抽检时使用。上线做 shadow/canary，监控分布漂移和线上反馈，再把失败 Trace 进入回归集。
- **深挖**：只看最终答案有什么问题？掩盖错误工具和危险尝试；Judge 与被测模型同源？错误相关；如何评副作用？在 sandbox/simulator 检查状态转换和不变量。
- **项目映射**：当前项目材料无完整评测运行记录，应讲设计而非冒充结果。
- **评分/危险**：单个 BLEU/准确率无法覆盖 Agent 行为。

## Q63. 如果面试官问项目中没实现的能力，怎么答？

- **来源映射**：所有项目深挖面经都在检验真实性；Q110/Q111进一步场景化。
- **30 秒答案**：先明确“当前没有实现”，再讲现状证据、为什么没做、可落地方案、代价和验证计划；不要偷换框架能力、依赖或设计文档为成果。
- **2 分钟标准答案**：使用五段式：事实范围 -> 当前调用链 -> 缺口原因/non-goal -> 改造设计 -> 验收指标。例如 ESGQA 没有通用 Tool Registry，就说明现有能力嵌在节点中；若开放 Agent，会先定义 ToolSpec、Policy 和 ToolResult，再迁移节点，并用参数错误、越权、副作用和回放测试验证。若面试官继续问数字，明确没有运行数据，不编造，并给评测设计。
- **深挖**：会不会显得项目弱？诚实边界和工程判断更可信；框架默认支持算实现吗？只有代码配置和运行证据才算；“我负责的不是这块”后怎么办？仍讲你核验到的边界和接口。
- **项目映射**：直接使用 [[project-evidence-map]] 五类标签。
- **评分/危险**：“我们后续准备做”不能用过去式写进成果。

## Q64. 用 5 分钟总结 Phase 1 的 Agent 工程观。

- **来源性质**：综合压轴题。
- **30 秒答案**：Agent 是受策略约束的状态化 Runtime：Loop 管控制，Tool 管能力，Router 管决策，Provider 管模型执行，Context 管有限信息，Security 管副作用边界。
- **2 分钟标准答案**：我先从目标和验收出发，把可确定流程留给 Workflow，把语义长尾交给受限 Agent。模型只产生候选动作，Runtime 维护 typed state，执行 Tool 前做 Schema、权限和预算校验，结果以 Observation 回注；Router 综合意图、能力、风险和成本；Provider 通过能力矩阵和事件协议适配；Context 在窗口内按价值、权限和来源选择；安全用 authn/authz/sandbox/HITL 纵深防御。全链路必须可观测、可取消、可恢复、有终止原因，并用组件+Trace+端到端评测。项目上，ESGQA 证明领域状态图和混合检索，agentHub 证明事件 Runtime、Provider、权限和压缩；所有未实现能力只作为改进建议。
- **深挖**：为什么不是 Prompt 工程观？控制权和副作用在 Runtime；为什么不是全用 ReAct？风险和可验证性；下一阶段补什么？模型/RAG 深度、生产评测、后训练与系统规模化。
- **项目映射**：至少引用一个 ESGQA 和一个 agentHub 证据，并主动给边界。
- **评分/危险**：罗列框架、论文和模型名，没有不变量与失败路径，不足以应对大厂追问。
