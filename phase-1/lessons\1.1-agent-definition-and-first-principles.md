# Lesson 1.1：Agent 定义与 First Principles

## 1. 课程定位、前置依赖与后续伏笔

本课不从“某框架的 `Agent()` API”开始，而从一个更严格的问题开始：**一个只会根据输入生成文本的概率模型，为什么以及在什么条件下会变成一个能持续完成任务的系统？**

前置依赖只有三点：理解 LLM 本质上执行条件生成；知道函数调用和外部 API 的基本概念；见过状态机的状态、转移和终止。下一课会把本课定义展开为 Agent Loop，后续 Tool、Router、Provider、Context 和 Security 都是这个闭环中的不同责任边界。

本课重点：

1. 把 LLM、Chatbot、Workflow、Agent 和自治系统放在同一条能力谱系上。
2. 从环境反馈和部分可观测性推导状态、行动、观察和终止为何不可缺失。
3. 学会判断“什么时候不应该用 Agent”。
4. 能用 ESGQA 与 agentHub 的真实代码证明边界，而不是背定义。

## 2. 学习目标、应答问题与课前诊断

完成本课后，你应能回答：

- 为什么“调用一次 LLM”不是 Agent？
- Agent 与确定性 Workflow 的边界在哪里？
- Tool Calling 是 Agent 的必要条件还是常见实现？
- 自主性越高越好吗？如何量化自主性？
- 为什么状态、预算、权限和终止条件属于 Agent 定义，而不是上线后的附加项？
- ESGQA 为什么更适合称为领域化 Agentic Workflow？
- agentHub 的 Hub Runtime 与底层 Agent SDK 各自负责什么？

课前诊断：如果你的回答只是“Agent = LLM + Tools”或“会自己思考的模型”，说明定义仍停留在组件罗列或拟人化层面。严格回答必须包含目标、环境、状态、策略、行动、反馈、约束和终止。

## 3. 一句话模型、问题动机、非目标与适用边界

> **一句话模型：Agent 是一个在目标与约束下，基于状态选择行动、接收环境反馈、更新状态并在明确条件下终止的闭环决策系统；LLM 只是其中一种可替换的策略实现。**

问题动机：LLM 的一次生成可写成 `y ~ Pθ(y | x)`。它接收固定上下文 `x`，输出 `y`，但它不会天然知道输出是否成功、不会自动访问最新外部世界、不会自动重试，也不会保证停止。复杂任务通常存在部分可观测、动态信息、外部副作用和中间失败，所以系统需要多轮“决定 -> 行动 -> 观察 -> 修正”。

非目标：

- 不把所有多步程序都叫 Agent。
- 不把隐藏的 Chain-of-Thought 当作可审计控制流。
- 不追求最大自主性；业务价值、风险和可验证性优先。
- 不在本课绑定 LangChain、LangGraph、OpenAI Agents SDK 或任何单一框架。

适用边界：当任务路径固定、输入结构稳定、错误代价高且可用规则完整时，普通函数或 Workflow 往往更好；当路径需要根据观察动态调整、工具集合有限且可以建立权限和终止边界时，Agent 才可能带来价值。

## 4. A 层完整推导：从概率生成到闭环决策

### 4.1 第一步：LLM 是策略候选，不是完整系统

将一次 LLM 调用抽象为：

\[
a_t \sim \pi_\theta(\cdot \mid h_t)
\]

其中 `h_t` 是当前可见历史，`a_t` 是文本、结构化决定或工具调用。它可以提出行动，但以下能力不由这个式子保证：

- `a_t` 是否被允许执行；
- 参数是否满足业务约束；
- 环境是否成功执行；
- 观察是否可信；
- 下一轮应该继续、重试、降级还是结束；
- 总成本是否超预算。

因此，**模型能力不能替代运行时责任**。

### 4.2 第二步：开放任务意味着环境反馈

若任务需要查询数据库、读文件、调用服务或修改状态，结果取决于环境 `E`。执行行动后得到观察：

\[
o_{t+1} = E(s_t, a_t)
\]

观察可能是成功数据，也可能是超时、权限拒绝、参数错误或部分成功。没有观察，系统无法判断行动效果；把工具输出直接当“事实”也不够，因为输出可能被攻击者污染或已经过期。

### 4.3 第三步：部分可观测要求显式状态

模型看不到完整环境，只看到选入上下文的消息和工具结果。运行时必须维护：

\[
s_{t+1} = U(s_t, a_t, o_{t+1})
\]

`s_t` 至少应包含任务目标、消息、已执行行动、预算、权限状态、错误、进展指纹和终止原因。只保存聊天历史会丢失机器可读不变量；只保存结构化状态又可能丢失用户语义，因此生产系统常同时维护“模型可见消息”和“运行时真实状态”。

### 4.4 第四步：目标不是 Prompt 中的一句话

目标 `g` 必须被拆成可验证的完成条件。例如“完成 CBAM 合规核算”至少包含：输入参数齐全、适用政策确定、计算过程可追踪、错误被显式报告、输出格式正确。若目标不可验证，Agent 只能凭语言上的“看起来完成”结束。

约束 `c` 包括权限、预算、时限、数据边界、允许工具和副作用审批。严格地说，Agent 每次选择的是可行动作集合：

\[
a_t \in A_{allowed}(s_t, c)
\]

不是模型生成什么就执行什么。

### 4.5 第五步：终止是闭环的一部分

终止谓词可以写成：

\[
Terminate(s_t) = GoalReached \lor MaxSteps \lor Deadline \lor BudgetExhausted \lor NoProgress \lor PolicyAbort
\]

没有上限的“让模型判断是否继续”不是可靠终止设计。成功和失败都必须是可表示的终态，否则系统会出现无限循环、静默失败或重复副作用。

### 4.6 可操作的五元组

沿用原大纲的五元组，可定义：

\[
Agent = (\pi_\theta, M, T, V, E)
\]

- `πθ`：决策策略，可以是 LLM、规则、分类器或混合策略。
- `M`：状态与记忆，包括运行态和模型上下文。
- `T`：可行动作/工具集合及其协议。
- `V`：验证、约束和终止判定。
- `E`：产生观察的外部环境。

目标 `g` 和约束 `c` 是一次运行的输入。若系统没有 `T`，它仍可能是只做路由或信息变换的 Agentic 系统；若没有环境反馈和状态更新，则通常只是单次模型调用。

### 4.7 最小控制流

```text
接收目标 g 与约束 c
初始化状态 s0
while not Terminate(st):
    xt = BuildContext(st)
    proposed = Policy(xt)
    decision = ValidateAndRoute(proposed, st, c)
    if decision is FINAL:
        验证答案并结束
    if decision is ACTION:
        检查权限和副作用
        observation = Environment.execute(action)
        st = Update(st, action, observation)
返回结构化终止原因、最终结果和轨迹
```

关键不变量：每轮只能由运行时推进 step；所有副作用先授权；每个 ToolCall 必有相关 ToolResult；预算单调消耗；终止一旦写入不可被后续模型文本覆盖。

## 5. 相似概念辨析、选型矩阵、反模式与边界条件

### 5.1 概念矩阵

| 系统 | 路径决定者 | 是否读取环境反馈 | 状态 | 典型用途 |
|---|---|---|---|---|
| LLM 单次调用 | 模型一次生成 | 否 | 请求上下文 | 改写、分类、摘要 |
| Chatbot | 对话轮次 + 模型 | 可选 | 对话历史 | 问答、陪伴、客服 |
| Workflow | 开发者预定义图/代码 | 是 | 显式业务状态 | 合规流程、审批、ETL |
| Agent | 策略在运行时选择下一行动 | 是 | 显式运行态 + 模型上下文 | 开放路径任务、探索与修复 |
| 自治系统 | Agent + 长期目标 + 持续运行 | 持续 | 持久状态 | 长周期运维或研究任务 |

Workflow 与 Agent 不是互斥集合。常见的生产形态是**外层确定性 Workflow，内层局部 Agent**：高风险阶段固定，只有确实需要动态决策的局部交给模型。

### 5.2 自主性谱系

可用四个维度判断，而不是问“是不是 Agent”：

1. **路径自主性**：下一步是固定还是运行时选择。
2. **工具自主性**：工具由代码固定调用还是模型选择。
3. **目标自主性**：只执行给定目标，还是能拆目标/创建子目标。
4. **持续自主性**：单次请求结束，还是跨会话持续运行。

自主性增加会同时增加搜索空间、不可预测性、成本、权限风险和验证难度。面试中若只讲“更智能”，通常会被追问击穿。

### 5.3 什么时候不使用 Agent

- 输入输出可以由确定性函数完整表达。
- 合规步骤固定，任何偏离都不可接受。
- 工具副作用高，但没有审批、幂等与补偿机制。
- 无法定义成功、失败和预算上限。
- 没有足够观测数据定位错误。
- 仅仅为了跟随热点，把一次 Prompt 包成循环。

### 5.4 高频反模式

- **Prompt 即系统**：所有状态和规则塞进一段 Prompt。
- **模型即权限系统**：用“请勿执行危险操作”代替代码检查。
- **模型自报成功**：没有外部验证就接受“任务完成”。
- **无限反思**：失败后不断让模型“再想想”，没有错误分类和上限。
- **过度 Agent 化**：固定业务链被改成自由规划，准确性下降且难以审计。

## 6. ESGQA 证据、调用链与事实边界

证据：`P1-ESG-STATEGRAPH-001`。

`CBAMState` 在 `agents.py:106-141` 显式保存 query、memory_context、task_type、核算参数、retrieved_context、error_msg 和 metrics；`build_cbam_graph()` 在 `agents.py:906-966` 注册节点和边。这满足本课中的状态、行动阶段、环境能力和终止分支，但路径大部分由开发者预定义。

**可说**：ESGQA 是带 LLM 分类、检索、计算、评估与失败分支的领域化 Agentic Workflow；它比单次 RAG 调用多了共享状态和控制流。

**不可说**：它已经实现通用自治 Agent、任意工具选择或通用 ReAct 循环。

面试调用链：

```text
用户查询
-> node_memory 组装记忆
-> node_intent_parser 结构化意图
-> route_intent 选择固定业务分支
-> 业务节点链
-> evaluator/export 或 fallback
```

严格评价：这不是“自主性不够所以落后”，而是风险与领域约束下的合理选择。CBAM 核算需要可解释和可审计，固定主链通常优于模型自由规划。

## 7. agentHub 证据、调用链与事实边界

证据：`P1-HUB-RUNTIME-001`。

`AgentRuntime.ts` 管理 Provider 创建、prompt 排队、事件处理、权限请求、压缩和空闲关闭。它体现的是系统级 Agent Runtime：目标和 prompt 到来后，Runtime 维持长期进程并接收 Provider 事件；具体模型推理与部分工具循环由 Provider SDK 处理。

**可说**：agentHub 已经把 Agent 从“一个模型调用”提升为有生命周期、队列、Provider、权限和会话资源的运行实体。

**不可说**：Runtime 自己实现了每一步 ReAct 推理，或 Hub 的所有能力都属于底层模型。

面试中要主动拆层：

```text
Hub 层：会话、任务、Agent 协作、上下文共享
Runtime 层：进程/容器、Provider、队列、事件、权限、idle close
Provider 层：对接 Claude Code/OpenCode SDK，产生统一事件
模型层：生成推理、文本或工具调用
```

这四层若混说，面试官只要问“代码在哪一层执行”就会暴露理解不清。

## 8. C 层扩展：生产架构、故障、成本、观测与安全

### 8.1 生产架构责任分离

一个可运维 Agent 至少需要：API/Queue、Runtime、Policy、Provider、Tool Gateway、State Store、Context Builder、Event Store、Evaluator。它们不必是独立微服务，但责任必须可测试、可替换和可观测。

### 8.2 故障分类

- 模型失败：限流、超时、上下文超限、畸形结构。
- 工具失败：参数、权限、超时、部分副作用。
- 业务失败：目标不可达、证据不足、规则冲突。
- 控制流失败：循环、状态不一致、重复执行。
- 基础设施失败：容器、队列、数据库、网络。

每类失败的重试语义不同。永久参数错误不应指数退避；有副作用的超时不能盲目重试；无进展应终止而非无限反思。

### 8.3 成本与延迟

总延迟近似为：

\[
L = \sum L_{model} + \sum L_{tool} + L_{queue} + L_{policy} + L_{serialization}
\]

总成本不仅是 Token，还包括外部 API、计算、人工审批和失败重做。自主循环每多一步都扩大尾延迟和失败概率。因此应设置 max_steps、deadline、token_budget、tool_budget 和 cost_budget。

### 8.4 观测

事件至少记录 run_id、step、parent_id、decision、tool_call_id、耗时、usage、结果状态和 termination_reason。Prompt/Tool 参数中可能有秘密，日志必须脱敏；可观测不等于把所有上下文原文写入日志。

### 8.5 安全

Agent 的核心安全原则是：**把模型输出视为不可信提议，而不是已授权命令。**权限、路径、网络、数据和副作用控制都必须由模型之外的执行边界实施。

## 9. 示例代码与状态推演（非运行实验）

> [!warning] 教学示例
> 以下代码未在本课程中运行，只用于展示类型边界。生产实现还需持久化、并发、超时、审计和安全测试。

```python
from enum import Enum
from typing import Any, Literal
from pydantic import BaseModel, Field

class TerminationCode(str, Enum):
    COMPLETED = "completed"
    MAX_STEPS = "max_steps"
    NO_PROGRESS = "no_progress"
    POLICY_ABORT = "policy_abort"

class Action(BaseModel):
    kind: Literal["respond", "tool"]
    tool_name: str | None = None
    arguments: dict[str, Any] = Field(default_factory=dict)

class AgentState(BaseModel):
    run_id: str
    goal: str
    step: int = 0
    observations: list[str] = Field(default_factory=list)
    termination: TerminationCode | None = None

def should_stop(state: AgentState, max_steps: int) -> bool:
    return state.termination is not None or state.step >= max_steps
```

面试讲解点：`Action` 是模型提议的结构，不是执行授权；`AgentState` 是运行时真相；`should_stop` 不能只问模型。示例没有 Policy、ToolResult 关联、预算和并发，所以只能说明契约，不能称为完整 Agent。

## 10. 文字 Demo：同一需求的三种系统实现

**Demo ID**：`DEMO-ESG-01` + `DEMO-HUB-01`。

**需求**：“根据材料判断一批钢材是否适用 CBAM，并给出成本估算。”

### 方案 A：单次 LLM

输入 Prompt 和材料，输出答案。优点是快；缺点是没有最新数据、工具验证、状态和失败出口。若模型编造碳价，系统无法察觉。

### 方案 B：ESGQA Workflow

状态变化：

```text
query
-> memory_context
-> task_type=calculation
-> retrieved_context
-> rates/factors
-> final_cbam_cost_eur
-> evaluator/export
```

异常分支：任一阶段设置 `error_msg`，条件边转 fallback。适合领域流程，因为关键阶段固定。

### 方案 C：开放 Agent Runtime

Runtime 让策略从受限工具中选择检索、读取政策或计算，并根据 ToolResult 决定下一步。它适合路径未知的任务，但必须额外实现工具 Schema、权限、预算和终止。

**一分钟讲解稿**：

“我不会因为系统用了 LLM 就称它为 Agent。我从闭环定义出发：目标、状态、策略、行动、观察、约束和终止。ESGQA 的主链是显式 StateGraph，所以我称它为领域化 Agentic Workflow；agentHub 则在更高层提供 Agent Runtime 生命周期。二者都说明 Agent 是系统属性，不是某个模型属性。生产选择上，CBAM 这类合规流程应让固定图承担主控制，只把意图识别或非结构化信息处理交给模型。”

## 11. 面试问题树

1. **定义题**：你如何定义 Agent？为什么 `LLM + Tools` 不够？
2. **First Principles**：从部分可观测环境推导状态与观察的必要性。
3. **源码题**：ESGQA 哪些代码证明它有闭环？哪些代码限制了自主性？
4. **架构题**：Workflow 与 Agent 如何混合，而不是二选一？
5. **故障题**：Agent 一直重复同一工具调用，如何定位和终止？
6. **反事实题**：若工具全部只读且任务无副作用，哪些安全措施仍不能省？
7. **成本题**：自主性增加为何会扩大尾延迟和评估难度？
8. **边界题**：agentHub 的 Runtime、Provider 和模型如何分层？
9. **选型题**：什么情况下宁愿用普通函数而不用 Agent？
10. **改造题**：如何把 ESGQA 从固定 Workflow 扩展为局部 Agent，又不破坏审计性？

压力追问：

- “你的定义是不是把任何带循环的程序都叫 Agent？”
- “状态都在聊天记录里，为什么还要结构化状态？”
- “模型能自己判断完成，为何还要终止代码？”
- “用了 LangGraph 就一定是 Agent 吗？”
- “你说自主性有风险，有没有可量化指标？”

## 12. 30 秒、2 分钟与深挖回答

### 12.1 30 秒版本

“我把 Agent 定义为目标与约束下的闭环决策系统：根据显式状态选择行动，从环境获得观察，更新状态，并由可验证条件终止。LLM 只是策略，Tool 只是行动通道。ESGQA 是固定主链的领域 Agentic Workflow；agentHub 的 Runtime 则管理 Provider、队列、权限和生命周期。是否使用 Agent取决于路径动态性是否值得承担更高的成本、风险和验证难度。”

### 12.2 2 分钟版本

先从 `P(y|x)` 说明单次模型没有环境反馈，再引出 `s -> a -> o -> s'`。强调目标、权限和终止不由模型自证。随后用 ESGQA 的 `CBAMState + StateGraph + fallback` 证明确定性 Workflow，用 agentHub `AgentRuntime + Provider + Event` 证明系统运行时。最后给选型结论：固定、高风险、可枚举流程用 Workflow；路径开放、需要根据观察调整且有受控工具时使用局部 Agent。

### 12.3 深挖评分点

- 能给出形式化状态转移，而非拟人化描述。
- 能区分模型策略、运行时、环境和权限。
- 能主动讲失败终态、预算和无进展。
- 能引用两个项目的文件/符号和事实边界。
- 能说明 Agent 与 Workflow 可组合。
- 不声称未运行的性能数字。

危险回答：

- “Agent 就是能自主思考的 LLM。”
- “LangGraph 项目都是 Agent。”
- “工具调用成功后模型自然知道下一步。”
- “自主性越高能力越强。”
- “项目已经实现通用 Agent”，却找不到 Tool Registry、Runtime 或终止逻辑。

## 13. 课后复习：口述提纲、闪卡、自测与易错点

### 13.1 五分钟口述提纲

1. 一次 LLM 调用缺什么。
2. `goal/state/policy/action/observation/constraint/termination`。
3. 五元组各自含义。
4. Workflow、Agent 与混合架构。
5. ESGQA 与 agentHub 各自证明什么。

### 13.2 闪卡

- Q：Agent 的最小闭环？ A：状态 -> 决策 -> 行动 -> 观察 -> 状态更新 -> 终止判断。
- Q：LLM 在 Agent 中是什么？ A：可替换的策略组件，不是权限或事实源。
- Q：为什么需要结构化状态？ A：维持不变量、预算、错误、关联 ID 和可重放控制流。
- Q：为什么模型不能自管权限？ A：模型输出不可信，且会受 Prompt Injection 影响。
- Q：ESGQA 的准确称谓？ A：CBAM 领域化 Agentic Workflow。
- Q：agentHub Runtime 的核心职责？ A：Provider、队列、事件、权限、压缩和生命周期。

### 13.3 自测

1. 设计一个“绝不该用 Agent”的业务并解释原因。
2. 为“调研竞品”定义至少五个终止条件。
3. 指出聊天历史与运行时状态各自不能替代对方的原因。
4. 把 ESGQA 的 calculation 主链画成状态转移，并标出 fallback。
5. 若 agentHub Provider 已经处理工具循环，Runtime 仍需负责什么？

### 13.4 易错点

- 把“有 LLM”误当“有 Agent”。
- 把 Tool Calling 当完整工具执行系统。
- 忽略失败终态和补偿。
- 把项目文档或依赖当实现。
- 用“自主思考”代替可测试的系统定义。

间隔复习建议仅供自愿使用：当天口述一次；第 3 天只看问题树回答；第 7 天用两个项目各讲一次；第 14 天做“为什么不用 Agent”的反向设计。
