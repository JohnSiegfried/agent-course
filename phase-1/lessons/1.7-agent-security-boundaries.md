# Lesson 1.7：Agent 安全边界与纵深防御

## 1. 课程定位、前置依赖与后续伏笔

Agent 安全的本质不是让模型更听话，而是即使模型被诱导、误判或输出恶意动作，受信任运行时仍把损害限制在可接受范围。Prompt 防护只是输入层；真正的安全来自身份、授权、数据边界、工具契约、沙箱、网络、密钥、审批、审计和恢复共同工作。

本课收束 Phase 1：Agent Loop 决定攻击者能否多步扩大影响；Tool-use 决定动作面和副作用；Router/Policy 决定允许、拒绝或审批；Provider 持有外部凭证；Context/Memory 可能携带不可信内容与敏感数据。

后续进入任何工程 Phase，都应把本课的 threat model 和权限不变量作为设计门，而不是上线前补一轮关键词过滤。

## 2. 学习目标、应答问题与课前诊断

完成后应能回答：

- Agent 系统的资产、主体、信任边界和攻击面有哪些？
- 直接/间接 Prompt Injection 为什么无法仅靠 Prompt 根治？
- Confused Deputy 如何发生？能力与授权如何分离？
- 文件路径校验为什么不能只做字符串 glob 或 prefix？
- symlink、TOCTOU、路径遍历如何进入 Agent 工具链？
- Docker 提供哪些隔离原语，又不自动保证什么？
- read-only root、capability drop、seccomp、rootless、网络限制分别解决什么？
- HITL 应在哪些动作生效，如何避免疲劳式批准？
- ESGQA 的 API 安全和 agentHub 的沙箱/权限各证明什么，不能证明什么？

课前诊断：如果你的安全答案是 system prompt 禁止访问敏感文件，再给容器跑，面试官会继续问容器挂载、用户身份、网络、Docker socket、工具 allowlist、路径解析、审批绑定和审计。任何一层答不上，都可能成为绕过路径。

## 3. 一句话模型、资产与信任边界

> **一句话模型：把模型及其接触的外部内容视为不可信决策输入，以最小权限、默认拒绝、强制策略、隔离执行、显式审批和可追溯审计限制真实副作用。**

### 3.1 资产

- 用户数据、企业文档、源码和凭证；
- 主机文件系统、进程、网络与云资源；
- Tool 能力和第三方 OAuth Token；
- 会话、记忆、向量索引和审计日志；
- 预算、GPU/API 配额和服务可用性；
- 系统 Prompt、策略规则和内部拓扑。

### 3.2 主体

- end user、tenant、管理员、开发者；
- Agent/模型、Runtime、Tool executor；
- Provider、MCP Server、网页/文档/仓库等外部来源；
- 容器内进程、宿主 daemon、后台 worker。

### 3.3 信任边界

```text
User / External Content：不可信
        |
API Gateway：认证、限流、输入大小
        |
Runtime + Policy Engine：受信任控制面
        | 只接受动作请求，不接受隐式授权
Tool Broker：Schema、授权、审批、审计
        |
Sandbox：文件、进程、网络、资源限制
        |
External Systems and Secrets
```

模型位于决策面，不是安全边界。它可以请求能力，不能自行定义权限，也不能把网页里的管理员已批准升级成真实授权。

## 4. 威胁模型：从 Prompt Injection 到权限扩大

### 4.1 Prompt Injection

- **直接注入**：用户明确要求忽略上层规则；
- **间接注入**：网页、PDF、issue、工具结果或记忆中包含对 Agent 的恶意指令；
- **持久注入**：恶意内容被写入长期记忆，后续会话继续触发；
- **跨 Agent 注入**：低信任 Agent 的输出被高权限 Agent 当指令。

内容分隔、来源标签和指令层级能降低误执行，但不能替代授权。外部文档即使自称来自管理员，也只是数据。

### 4.2 Confused Deputy 与 Excessive Agency

Runtime 拥有用户没有的凭证，模型被诱导代替攻击者调用高权限工具，就形成 Confused Deputy。防护重点：

- 每次动作绑定真实 actor、tenant、purpose 和 resource；
- Tool Broker 按请求重新授权，不继承模型声称的批准；
- 高风险动作最小 scope、短期令牌、必要时人审；
- 授权结果绑定参数摘要，参数变化后必须重新审批。

### 4.3 数据泄露

泄露路径包括 Prompt/Provider、工具结果、日志、trace、错误栈、缓存、向量库、跨租户检索和网络外传。数据分类应在进入模型前生效，并决定可用 Provider、地域、保留期和脱敏策略。

### 4.4 文件、命令、网络与供应链

文件风险包括路径遍历、绝对路径、大小写和设备路径差异、symlink 跳出根目录、校验后替换目标的 TOCTOU。命令风险包括参数注入、环境变量与配置泄露、递归写删除扩大影响。仅匹配 `workspace/**` 不足以证明安全：glob 处理字符串，不一定解析真实路径、符号链接和打开时竞态。

默认开放外网会让恶意内容外传数据或拉取未审计代码。应按任务配置 egress allowlist、DNS/代理、大小和时间限制。依赖、镜像、MCP Server、插件和索引文件都需要来源、固定版本、签名或哈希与漏洞治理。

## 5. 纵深防御：身份、策略、审批、沙箱与审计

### 5.1 入口控制不等于动作授权

身份认证回答是谁；限流回答多久调用多少；动作授权回答这个主体现在能否对这个资源做这个动作。JWT 通过不等于可以执行任意 Tool。

### 5.2 默认拒绝与能力最小化

按角色和任务只暴露必要工具；按参数限制资源范围；读写分离；高风险动作使用临时 capability token。未知工具、未知路径、未知网络目标默认拒绝。空 allowlist 的语义必须明确并测试。

### 5.3 HITL 的正确位置

不可逆删除、资金或外发、权限变化、敏感数据出域、运行未知代码、扩大网络/文件范围应进入审批。审批 UI 展示具体动作、规范化参数、目标、风险、diff/影响和有效期；批准绑定动作摘要，参数变化即失效。

低风险读操作不应全部弹窗，否则形成无脑批准。Trust 模式必须有受限范围、撤销、到期和审计。

### 5.4 沙箱分层

容器提供 namespace、cgroup 等隔离原语，但安全取决于运行配置和宿主边界。评审至少包括：

- 非 root 用户与适用时的 rootless daemon；
- 只读根文件系统，仅挂载必要目录；
- 精确读写 mount，避免宿主敏感路径；
- drop capabilities 与 no-new-privileges；
- 默认 seccomp，并结合 AppArmor 或 SELinux；
- CPU、内存、PID、磁盘、时间限制；
- 默认无网络或 egress allowlist；
- 不向不可信容器暴露 Docker socket；
- 镜像固定 digest、最小基础镜像和漏洞扫描。

容器不能自动防御共享内核漏洞、错误挂载、daemon 权限和宿主凭证暴露。安全结论必须基于完整配置。

### 5.5 审计与事故响应

记录 actor、tenant、session/run、模型与规则版本、Tool、规范化参数摘要、授权结果、审批人、资源、sandbox ID、网络目标、结果、错误与时间。日志自身要脱敏、访问控制和防篡改。系统还需要 kill switch、凭证吊销、会话终止、隔离 artifact、通知和复盘流程。

## 7. agentHub 证据、调用链与事实边界

证据：`P1-HUB-SANDBOX-001`、`P1-HUB-PERMISSION-001`、`P1-HUB-DOC-SECURITY-GAPS-001`。

`SandboxManager.ts` 为会话创建 Docker 容器，设置 `/sandbox`、`/workspace`、`/home/agents` 挂载和内存等参数。`PermissionProfiles.ts` 定义角色工具与写路径，Runtime 发出权限事件。

可以严格陈述：

- **[已实现]** per-session Docker 生命周期；
- **[已实现]** 工作区和 Agent 目录挂载与资源参数；
- **[已实现]** 角色级工具、可写路径配置和权限事件；
- **[限制]** 有容器与配置不等于所有逃逸和越权路径已封闭。

必须记住的源码语义：`allowedTools` 仅在列表非空时执行包含检查，因此空列表不能自然解释为拒绝所有；未知或自定义角色存在较宽的 `**/*` 写模式；路径判断使用 glob，不能单独证明 canonical path、symlink 和 TOCTOU 安全。

改进应把策略显式化为 `deny_all / allow_list / allow_all`，默认拒绝；路径授权使用受控根、打开时不跟随链接的系统原语和精确 mount。项目 threat model 还明确记录未启用 seccomp/AppArmor、只读根文件系统，Docker socket、iframe sandbox/CSP、Trust ON 审计和 workspace 写边界等缺口。它证明团队识别了风险，不证明风险已经修复或项目通过外部安全认证。


## 8. 教学代码示例：受控根目录与审批绑定

> [!warning] 教学代码边界
> 以下代码只说明先规范化并绑定审批的思路，未在两个项目中运行，也不是跨平台完整安全实现。真实文件安全还需处理 symlink、重解析点、TOCTOU、平台 API、容器挂载与并发攻击。

```python
from dataclasses import dataclass
from hashlib import sha256
from pathlib import Path

@dataclass(frozen=True)
class ApprovedAction:
    actor: str
    operation: str
    canonical_target: str
    request_hash: str

def resolve_under_root(root: Path, requested: str) -> Path:
    root = root.resolve(strict=True)
    target = (root / requested).resolve(strict=False)
    try:
        target.relative_to(root)
    except ValueError as exc:
        raise PermissionError('TARGET_OUTSIDE_ALLOWED_ROOT') from exc
    return target

def action_hash(actor: str, operation: str, target: Path,
                content_digest: str) -> str:
    payload = '\n'.join([actor, operation, str(target), content_digest])
    return sha256(payload.encode('utf-8')).hexdigest()

def authorize_write(actor: str, root: Path, requested: str,
                    content_digest: str,
                    approval: ApprovedAction) -> Path:
    target = resolve_under_root(root, requested)
    expected = action_hash(actor, 'write', target, content_digest)
    if approval.request_hash != expected:
        raise PermissionError('APPROVAL_DOES_NOT_MATCH_ACTION')
    return target
```

生产改进：使用支持相对已打开目录句柄并拒绝跟随链接的系统调用；写临时文件后原子替换；限制类型和大小；操作前后验证 handle；Tool Broker 与 sandbox 双重限制。单次 `resolve()` 不能消除校验后到使用前的竞态。

## 10. 常见错误、反例与排障路径

1. **只靠 Prompt 禁止危险行为。** 模型不是授权边界。
2. **认证等于授权。** JWT 通过仍需资源级策略。
3. **空 allowlist 语义不明确。** 默认行为必须显式和测试。
4. **路径 prefix 或 glob 即安全。** 忽略规范化、链接和 TOCTOU。
5. **Docker 即安全。** 忽略 socket、root、mount、网络、capability 和共享内核。
6. **所有 Tool 都让用户批准。** 导致审批疲劳，应按风险分层。
7. **审批后参数可变化。** 必须绑定规范化动作摘要。
8. **fallback 放宽 Provider 或工具。** 安全不变量必须跨降级保持。
9. **日志记录所有 Prompt。** 可能二次泄露密钥和敏感文档。
10. **只防输入，不管记忆。** 持久注入会跨会话复发。
11. **安全异常自动重试。** 权限拒绝和认证失败不应被重试绕过。
12. **文档列出缺口等于已修复。** 必须找到配置和验证证据。

排障顺序：冻结 run、actor、tenant -> 吊销凭证并停止工具 -> 保存脱敏审计证据 -> 重放授权与审批 -> 检查真实路径、mount、network -> 定位注入来源和记忆传播 -> 评估影响范围 -> 修策略、隔离和测试 -> 复盘与轮换密钥。

## 11. 面试追问树、回答骨架与危险答案

### 主问题：如何保护能执行工具的 Agent？

30 秒回答：把模型和外部内容当不可信；Tool Broker 做 Schema、actor-bound authz 和风险审批；任务在最小权限沙箱运行，文件、网络和资源受限；全链路审计并支持 kill switch。Prompt 只做辅助防护。

2 分钟展开：

1. 资产、主体、信任边界和威胁矩阵；
2. Prompt Injection 与 Confused Deputy；
3. default deny、capability scope、HITL 参数绑定；
4. canonical path、symlink、TOCTOU 与命令 API；
5. Docker mount、root、cap、seccomp、network、socket；
6. 密钥、数据分类、日志和跨租户；
7. 红队、配置测试、审计与事故响应；
8. 用项目证据说明已实现能力和缺口。

追问：能不能彻底消灭 Prompt Injection？

答：不能把自然语言层保证当成绝对安全。目标是降低诱导成功率，并通过权限、隔离、审批和审计使成功诱导也无法越权。

追问：容器里运行就够了吗？

答：不够。要审计用户、mount、root/capabilities、seccomp/LSM、网络、资源、镜像、socket 和宿主；容器只是隔离原语之一。

追问：如何测权限？

答：正负权限矩阵、未知角色、空列表、路径遍历、绝对路径、链接、并发替换、参数变化后审批失效、Provider fallback、网络 egress、容器配置断言和审计完整性。

危险答案：

- 模型不会执行不安全命令，因为 system prompt 禁止了。
- 有 JWT，所以用户数据安全。
- Docker 和 glob 已经解决文件安全。
- 让用户每次点允许即可。
- 威胁模型写了这些项，所以都已经完成。

## 12. 大厂级安全评审清单与证据口径

- [ ] 有资产、主体、入口、边界和滥用场景，不只列漏洞名。
- [ ] 模型与外部内容按不可信处理，指令和数据边界清晰。
- [ ] authn、business authz、tool authz 三层不混淆。
- [ ] 工具最小暴露，未知主体、工具、资源默认拒绝。
- [ ] allowlist 空值与缺省值有明确语义和负向测试。
- [ ] 高风险审批绑定 actor、动作、参数、资源和有效期。
- [ ] 文件访问处理 canonical path、symlink/reparse 与 TOCTOU。
- [ ] 命令使用结构化 argv，限制可执行文件、环境和工作目录。
- [ ] sandbox mount 最小、rootfs 只读、非 root、cap drop、no-new-privileges。
- [ ] seccomp/LSM、网络 egress、PID/CPU/memory/disk/deadline 有配置证据。
- [ ] 不可信执行环境不暴露 Docker socket 和宿主凭证。
- [ ] secrets 由受控 broker 注入且短期、最小 scope、可吊销。
- [ ] Provider、memory、cache、vector store、logs 有 tenant 和数据分类边界。
- [ ] 重试与 fallback 不扩大权限或重复副作用。
- [ ] 审计日志可关联 actor、run、tool、approval、result，且自身受保护。
- [ ] 有 kill switch、凭证轮换、隔离、通知和复盘流程。
- [ ] 每项安全主张能指向代码、部署配置、测试或明确文档状态。

## 13. 复习卡、口述任务与自我检查

### 必背卡

- 模型不是安全边界；外部内容是不可信数据。
- authn != authz != tool execution isolation。
- 路径安全：规范化、受控根、链接、TOCTOU、mount，缺一不可。
- 容器安全：user/root、mount、cap、seccomp/LSM、network、resource、socket。
- 审批：风险分层，绑定具体参数，变化即失效。
- 项目：ESGQA 有 API 入口安全但无 Agent 沙箱；agentHub 有 per-session Docker 和权限，但 threat model 列有明确缺口。

### 口述任务

1. 用 3 分钟从零画 Agent threat model。
2. 解释为什么 `allowedTools=[]` 必须读实现而不能猜。
3. 回答用了 Docker 为什么还不安全，至少列 7 个检查面。
4. 用两个项目各讲一项已实现、一项缺口、一项优先改造。

### 自我检查答案

问：为什么 Prompt Injection 与 SQL Injection 不完全同类？答：自然语言指令与数据边界天然模糊，无法只靠转义建立同等强边界；必须限制后续能力和授权。

问：为什么审批要绑定动作摘要？答：防止用户批准 A 后模型或竞态把参数替换成 B。

问：为什么 Docker socket 风险高？答：能控制 daemon 往往意味着可以创建高权限容器或挂载宿主资源，破坏原有边界。
