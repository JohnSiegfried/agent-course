# Lesson 1.5：Provider 抽象与多模型运行时

## 1. 课程定位、前置依赖与后续伏笔

Router 决定“选谁”，Provider Adapter 负责“如何以这个 Provider 的真实语义执行”。一个成熟抽象不追求把所有 SDK 压成完全相同，而是在稳定最小接口上保留能力差异、错误语义和逃生口。

本课覆盖模型/Agent SDK Provider，不把向量数据库、对象存储等基础设施 Provider 混进同一接口。它与三课直接相连：

- Tool-use：不同 Provider 对工具定义、调用 ID、结果回传和并行调用语义不同；
- Router：Provider 能力、健康、成本和延迟成为路由输入；
- Context：Provider 的窗口、Usage 与消息格式影响预算；
- Security：API Key、网络出口、审批和工具权限不能因切换 Provider 被绕过。

## 2. 学习目标、应答问题与课前诊断

完成后应能回答：

- 最小 Provider 接口包含什么？为什么不能只有 `chat(messages)`？
- capabilities 如何声明、协商和缓存？
- 如何统一 streaming 事件又不丢失 Provider 特有语义？
- tool call、tool result、usage、finish reason 如何规范化？
- timeout、cancel、retry、fallback、circuit breaker 如何分层？
- 如何区分可重试限流、认证错误、内容策略拒绝和上下文溢出？
- 多 Provider fallback 如何避免语义漂移、重复副作用和权限扩大？
- ESGQA 和 agentHub 当前的真实实现边界分别是什么？

课前诊断：若切换模型只需换一个 model 字符串，你可能只是同一 SDK 内换模型；真正的跨 Provider 需要处理消息、工具、流式、会话、错误和可观测性差异。

## 3. 一句话模型、最小公分母与抽象泄漏

> **一句话模型：Provider Adapter 把厂商 SDK 的生命周期和事件翻译成运行时稳定契约，同时通过 capability 与原生扩展显式保留无法统一的差异。**

一个实用接口通常覆盖：

```text
describe_capabilities()
start(session/config)
send(request)
stream(request) -> ProviderEvent*
cancel(run_id)
resume(session_id, input)
stop()
normalize_error(error)
```

核心对象：

- `ProviderRequest`：规范化消息、工具、输出约束、预算、deadline、metadata；
- `ProviderEvent`：text delta、thinking/analysis 状态、tool call、tool result、subagent、usage、error、completed；
- `ProviderUsage`：输入、输出、缓存、推理等可获得计量，并保留“未知”；
- `ProviderCapabilities`：streaming、tools、parallel tools、structured output、session resume、cancel、max context 等；
- `ProviderError`：可用于运行时决策的稳定错误分类。

抽象泄漏不是失败，而是现实：Provider A 的会话恢复可能依赖远端 ID，Provider B 只有无状态消息；某家支持并行工具，另一家只支持串行；Usage 字段粒度不同。错误做法是假装都一样，正确做法是能力发现、明确降级并保留 `native_metadata`。

## 4. 能力、消息、工具与事件规范化

### 4.1 Capability Matrix

不要根据 Provider 名称写散落的 `if provider == ...`。注册时返回能力矩阵：

| 能力 | 类型 | 运行时用途 |
|---|---|---|
| `streaming` | bool | 是否允许增量 UI 与早停 |
| `tool_calling` | enum | none/single/parallel |
| `structured_output` | enum | none/prompt/schema/strict |
| `session_resume` | bool | 是否可恢复远端会话 |
| `cancellation` | bool | 是真取消还是仅停止消费 |
| `usage_detail` | set | 可获得哪些 Token/缓存字段 |
| `context_limit` | int/unknown | 预算预检，不写死为永久事实 |

能力应带 Provider/Adapter 版本，避免升级 SDK 后旧缓存继续误判。

### 4.2 消息规范化

内部消息可以统一为 `role + typed content parts + metadata`，但映射时要处理：

- system/developer 指令层级差异；
- 文本、图片、文件和音频的内容块；
- assistant tool call 与 tool result 的配对；
- 不同 Provider 对空内容、名称和顺序的限制；
- 历史 thinking 是否可回传、是否应持久化；
- Provider 会话 ID 与本地 trace/session ID 的区分。

不可逆转换要有告警。例如 Provider 不支持某种内容块时，不应静默丢弃后继续执行高风险任务。

### 4.3 工具规范化

内部 ToolSpec 到厂商格式的映射至少验证：name 长度/字符、JSON Schema 子集、描述长度、required/default、strict 支持、并行调用与审批语义。`tool_call_id` 必须在响应与结果之间稳定映射。

MCP 是 Host/Client/Server 连接和能力协商协议；Provider Adapter 是模型或 Agent SDK 的执行适配层。MCP 工具最终可以由 Provider 暴露给模型，但两者不是同一个抽象。

### 4.4 事件规范化

流式事件应有单调序号和稳定 run ID：

```text
run.started -> message.delta* -> tool.requested
-> tool.approval_required? -> tool.completed
-> usage.updated* -> run.completed | run.failed | run.cancelled
```

要允许 Provider 特有事件进入 `native` 通道，但业务关键状态不能只藏在原始事件里。运行时应验证非法序列，例如 completed 后继续 delta、同一 tool call 重复 completed、缺少 terminal event。

## 5. 生命周期、错误分类、重试与降级

### 5.1 deadline 与取消

timeout 是运行时不再等待；cancel 是通知 Provider 停止工作；二者不等价。若 SDK 不支持真取消，Adapter 应标记 `cancellation: best_effort`，并防止迟到事件改变已经终止的状态。

所有子动作继承绝对 deadline，比每层各设 30 秒更可靠。否则三层重试可能把用户请求拖到数分钟。

### 5.2 稳定错误分类

建议至少包括：

- `AUTHENTICATION`：不可重试，告警并阻断；
- `PERMISSION`：不可通过换 Provider 绕过；
- `RATE_LIMIT`：读取 retry-after，延迟或路由；
- `TRANSIENT_NETWORK`：有界退避；
- `PROVIDER_UNAVAILABLE`：熔断候选；
- `INVALID_REQUEST`：修正请求，不盲重试；
- `CONTEXT_OVERFLOW`：进入压缩/裁剪流程；
- `CONTENT_POLICY`：按产品政策处理，不应伪装网络故障；
- `TOOL_PROTOCOL`：修复调用/结果配对；
- `UNKNOWN`：保守终止并保留原始诊断。

错误规范化同时保留 `provider_code`、HTTP 状态、request ID 和经过脱敏的原始信息。

### 5.3 retry 与幂等

只对明确可重试错误进行指数退避并加 jitter；遵守 deadline、最大尝试次数和 Provider 的 retry-after。模型生成请求通常会产生新输出，业务层不能把它当作严格幂等；若上一次已发起工具副作用，重试 Provider 前必须确认 tool call 的提交状态。

### 5.4 circuit breaker 与健康

以 Provider/模型/区域/错误类为维度维护 closed/open/half-open，而不是一个全局布尔值。短时限流和持续认证失败不能共用策略。健康信号进入 Router 时要有时间戳，避免使用过期状态。

### 5.5 fallback 的不变量

切换 Provider 必须保持：权限不扩大、工具集合不扩大、系统指令不降级、输出契约可满足、数据驻留合规、预算可接受、已发生副作用不重复。无法保持时应拒绝 fallback。

## 6. ESGQA 证据、调用链与事实边界

证据：`P1-ESG-GAP-PROVIDER-001`。

冻结代码中各节点直接构造或调用特定模型集成（例如 `ChatOllama`），没有发现统一 Provider 接口、Factory、能力矩阵、标准事件或跨 Provider 错误协议。

严格表述：

- **[已实现]** 业务节点能够调用当前选定的模型集成；
- **[可演示]** 该模型参与意图解析、结构化输出或回答链；
- **[未实现]** 通用多 Provider Adapter 和运行时 Factory；
- **[未实现]** 跨 Provider 流式事件、Usage、取消和错误规范化；
- **[未实现]** 基于成本、能力、健康与合规的 Provider 路由。

不能说“换一个配置就支持任意模型”。即使 LangChain 提供多个集成，项目源码没有形成自己的稳定运行时契约，业务节点也可能依赖特定模型行为。

建议的演进顺序：

1. 盘点节点真正依赖：结构化输出、流式、上下文、工具；
2. 抽出最小 `ModelGateway` 与错误/Usage，不先承诺全功能统一；
3. 写当前 Ollama Adapter，使用契约测试锁住行为；
4. 再接第二 Provider，暴露抽象泄漏并修订能力矩阵；
5. Router 只选择满足所需能力与策略的候选。

## 7. agentHub 证据、调用链与事实边界

证据：`P1-HUB-PROVIDER-001`、`P1-HUB-GAP-CODEX-PROVIDER-001`、`P1-HUB-RUNTIME-001`。

冻结源码中的主链：

```text
ProviderFactory.init()
  -> 注册 claude-code / opencode / test
ProviderFactory.create(name)
  -> AbstractProvider.start()
  -> sendPrompt()
  -> onEvent() 输出统一事件
  -> stop()
AgentRuntime 消费事件、维护 busy/queue/permission/compression 状态
```

`providers/base.ts` 提供抽象接口和事件结构，覆盖 thinking、tool、subagent、text、usage 等类别；Factory 对未知名称报错。可以据此陈述：

- **[已实现]** Runtime 依赖 Provider 接口，不直接绑定单一具体实现；
- **[已实现]** Factory 注册并创建 `claude-code`、`opencode` 和测试 Provider；
- **[已实现]** Provider 事件被 Runtime 统一消费；
- **[已实现]** Runtime 对启动、发送、繁忙队列、权限事件、压缩和停止有生命周期处理。

关键边界：

- **[未实现]** 冻结提交中没有 `providers/codex.ts`，Factory 也未注册 Codex；
- **[事实边界]** 依赖清单出现 Codex SDK 只说明依赖存在，不说明调用链已接通；
- **[需改进]** 统一事件仍可能漏掉 SDK 特有会话、工具、取消与 Usage 语义；
- **[需改进]** 应增加 provider contract test、capability matrix、稳定错误类和 fallback 不变量测试。

面试时主动承认 Codex 缺口反而更可信：说明你会沿现有 `AbstractProvider + Factory + Runtime event` 扩展，而不是只“调通一次 API”。

## 8. 教学代码示例：能力感知 Provider 契约

> [!warning] 教学代码边界
> 以下是接口设计示例，未在 ESGQA 或 agentHub 中运行，不代表项目已有这些类型、重试或熔断实现。

```typescript
type ErrorKind =
  | "authentication" | "permission" | "rate_limit"
  | "transient_network" | "context_overflow"
  | "invalid_request" | "provider_unavailable" | "unknown";

interface Capabilities {
  streaming: boolean;
  toolCalling: "none" | "single" | "parallel";
  structuredOutput: "none" | "schema" | "strict";
  cancellation: "none" | "best_effort" | "supported";
  sessionResume: boolean;
}

type ProviderEvent =
  | { type: "text_delta"; runId: string; seq: number; text: string }
  | { type: "tool_requested"; runId: string; seq: number; callId: string; name: string; input: unknown }
  | { type: "usage"; runId: string; seq: number; inputTokens?: number; outputTokens?: number }
  | { type: "completed"; runId: string; seq: number; finishReason: string }
  | { type: "failed"; runId: string; seq: number; kind: ErrorKind; retryable: boolean };

interface Provider {
  readonly name: string;
  capabilities(): Promise<Capabilities>;
  stream(request: NormalizedRequest, signal: AbortSignal): AsyncIterable<ProviderEvent>;
  normalizeError(error: unknown): { kind: ErrorKind; retryable: boolean; providerCode?: string };
}

async function selectProvider(required: Partial<Capabilities>, candidates: Provider[]) {
  const checked = await Promise.all(candidates.map(async p => ({ p, c: await p.capabilities() })));
  const match = checked.find(({ c }) =>
    (!required.streaming || c.streaming) &&
    (!required.sessionResume || c.sessionResume)
  );
  if (!match) throw new Error("NO_CAPABLE_PROVIDER");
  return match.p;
}
```

示例故意不写自动 retry：是否重试需要 error kind、deadline、请求阶段和副作用状态共同决定，不能塞进一个无条件装饰器。

## 10. 常见错误、反例与排障路径

1. **只统一 `generate()`。** 一遇流式、工具和会话就散落厂商分支。
2. **把未知 Usage 写成 0。** 成本统计会被系统性低估；应使用 `unknown/null`。
3. **所有 4xx 都不重试，所有 5xx 都重试。** 需要结合 Provider code、请求阶段和 retry-after。
4. **fallback 扩大工具权限。** 候选 Provider 必须继承同一策略约束。
5. **停止消费流等于真取消。** 远端仍可能计费或继续触发事件。
6. **把 model 名当 capability。** 能力随 SDK、账户、区域和配置变化。
7. **静默丢弃不支持字段。** 应明确降级、拒绝或暴露告警。
8. **Adapter 内完成业务路由。** Provider 只负责语义适配，策略选择属于 Router/Runtime。
9. **重试导致工具副作用重复。** 必须关联 run/tool call 状态和幂等键。
10. **依赖存在即宣布支持。** 必须找到注册、构造、调用、事件和测试闭环。

排障顺序：确认 normalized request -> 检查 capability snapshot -> 检查 Adapter 映射 -> 按 seq 重放事件 -> 查看 normalized error 与原始 request ID -> 检查 Runtime retry/fallback -> 核对 Usage 和最终状态。

## 11. 面试追问树、回答骨架与危险答案

### 主问题：如何设计多模型 Provider 层？

30 秒回答：定义最小稳定接口、结构化事件和错误；用 capability matrix 表达差异；Router 依据能力/健康/成本选择；fallback 保持权限和输出不变量；厂商特性通过扩展字段保留。

2 分钟展开：

1. request/message/tool/usage/event 规范化；
2. start/stream/cancel/resume/stop 生命周期；
3. 错误 taxonomy、deadline、有界 retry、熔断；
4. contract test 与 replay 测试；
5. 可观测性和成本；
6. 项目举证：agentHub 已有抽象与两个生产注册项，ESGQA 尚无通用层，Codex 也未接通。

追问：统一接口会不会抹平高级能力？

答：所以采用“稳定核心 + capabilities + provider extensions”，调用方明确声明 required capability；无法无损映射时拒绝或显式降级。

追问：如何新增 Codex Provider？

答：先按现有 `AbstractProvider` 实现生命周期和事件映射，再接 Runtime 的会话、工具、权限、取消和 Usage；注册 Factory；最后以同一 contract suite 验证正常、错误、取消、迟到事件和队列行为。

危险答案：

- “所有 OpenAI-compatible 接口都一样。”
- “失败就切下一个模型。”
- “SDK 会自动处理所有重试。”
- “停止前端流就取消了请求。”
- “agentHub package.json 有 Codex，所以已经支持。”

## 12. 大厂级设计检查清单与评审口径

- [ ] Provider 接口覆盖生命周期、流式、取消和终态。
- [ ] capability 可发现、带版本且不依赖名称猜测。
- [ ] 消息转换不会静默丢字段或改变指令层级。
- [ ] tool call/result ID 和并行语义稳定映射。
- [ ] 事件有 run ID、seq、terminal invariant 和迟到事件处理。
- [ ] Usage 的缺失值不会伪装为 0。
- [ ] 错误有稳定分类并保留脱敏原始诊断。
- [ ] retry 遵守 retry-after、deadline、幂等和尝试预算。
- [ ] circuit breaker 维度合理，并有 half-open 探测。
- [ ] fallback 不扩大权限、工具、数据地域或成本。
- [ ] Provider API Key 不进入 Prompt、日志或前端。
- [ ] contract tests 覆盖两个以上实现，证明抽象不是单实现自说自话。
- [ ] 原生能力有受控 escape hatch，不污染通用业务层。
- [ ] 项目表述与冻结版本一致，未把依赖冒充实现。

## 13. 复习卡、口述任务与自我检查

### 必背卡

- 抽象策略：稳定核心 + capabilities + extensions。
- 事件：started/delta/tool/usage/completed|failed|cancelled，有序且唯一终态。
- 错误：认证、权限、限流、瞬时、不可用、非法请求、上下文溢出、策略、协议、未知。
- fallback 不变量：权限、工具、指令、输出、合规、副作用、预算。
- 项目事实：agentHub 注册 claude-code/opencode/test；无 Codex Provider。ESGQA 没有通用 Provider 层。

### 口述任务

1. 用 2 分钟解释为什么 `chat(messages)` 不够。
2. 画出 Provider Event 到 Runtime State 的转换。
3. 分别解释 timeout、cancel、retry、fallback、circuit breaker。
4. 回答“如何证明新增 Provider 真正接通”并列出证据链。

### 自我检查答案

问：为什么 usage 也属于 Provider 契约？答：它驱动预算、成本归因、压缩和路由；不同来源需规范化并保留未知。

问：MCP 和 Provider 有何区别？答：MCP 连接能力提供方与 Host；Provider Adapter 适配模型/Agent SDK 执行语义，二者可组合但维度不同。

问：为什么 contract test 比 mock 单元测试重要？答：它用同一行为套件约束多个真实 Adapter，暴露规范化假设和抽象泄漏。
