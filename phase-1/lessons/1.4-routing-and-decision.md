# Lesson 1.4：Router 与决策边界

## 1. 课程定位、前置依赖与后续伏笔

Tool 解决“系统允许做什么”，Router 解决“当前应该走哪条路径”。两者若混在一起，模型的一次分类输出就可能同时决定业务流程、权限与高风险副作用，难以测试也难以追责。

本课不把 Router 限定为“意图分类器”。生产路由至少包含四类决策：

1. **任务路由**：问答、报告、指标、闲聊、拒绝等业务路径。
2. **能力路由**：选择 Tool、Agent、Provider 或执行队列。
3. **策略路由**：允许、拒绝、降级、人工审批。
4. **故障路由**：重试、fallback、补偿、终止。

前置依赖是 1.2 的控制流与 1.3 的工具契约。下一课会把 Provider 作为一种可路由但不能被过度统一的能力；1.7 会进一步说明安全策略不能只交给模型 Router。

## 2. 学习目标、应答问题与课前诊断

完成本课后，应能在白板上回答：

- 规则路由、模型路由、策略路由各适合什么场景？
- 如何定义一个可审计的 `RouteDecision`，而不是只返回字符串？
- confidence 为什么不是天然概率？阈值如何校准？
- 多条规则同时命中时，优先级、specificity 和默认规则如何处理？
- fallback、reject、HITL 和 retry 有何差别？
- 如何发现 unreachable route、循环路由和默认分支吞错？
- Router 的质量应看准确率，还是业务损失、成本与延迟？
- ESGQA 与 agentHub 的路由分别位于哪一层？源码没有实现哪些生产能力？

课前诊断：如果你的 Router 只有 `if label == "x"`，没有理由、版本、来源、策略结果和默认行为，那么即使分类准确率很高，也很难解释一次错误动作为什么发生。

## 3. 一句话模型、问题动机、非目标与适用边界

> **一句话模型：Router 是把当前状态映射为显式、可验证、可审计决策的控制面；模型可以提供候选判断，但规则和策略层保留最终执行权。**

可抽象为：

\[
d = R(s, c, p, v)
\]

- \(s\)：业务与运行状态；
- \(c\)：候选能力及其能力声明；
- \(p\)：权限、预算、合规和风险策略；
- \(v\)：路由规则与模型版本；
- \(d\)：结构化决策，而非裸标签。

建议的结果结构至少包含：

```text
RouteDecision = {
  route, reason_code, confidence,
  matched_rule, policy_effect,
  fallback, requires_approval,
  router_version, trace_id
}
```

非目标：Router 不负责真正执行工具，不替代 Tool 参数验证，不承担完整规划，也不等同于负载均衡。它可以选择 Planner，但不应该暗中完成整个 Plan。

## 4. 核心原理：四层路由与决策优先级

### 4.1 第一层：确定性规则

规则适合高确定性、高风险、可枚举条件：认证失败、预算超限、敏感操作、明确命令、Provider 不支持某能力。优点是可解释、低延迟、稳定；缺点是维护成本、规则冲突和长尾覆盖不足。

典型顺序：

1. 硬拒绝与系统安全规则；
2. 明确业务规则；
3. 模型分类补充长尾；
4. 低置信度澄清或人工审批；
5. 默认安全终止。

这里的“安全规则优先”不是让安全代码散落在 Router，而是让独立 Policy Engine 的结果拥有否决权。

### 4.2 第二层：模型路由

模型适合自然语言语义、开放类别和规则难以穷举的输入。输出必须结构化，并限定候选集合。Schema 只验证形状，仍需检查：

- `route` 是否属于当前可用候选；
- 输入是否具备该路由的前置条件；
- 理由是否与证据一致；
- confidence 是否经过离线校准；
- 高风险路线是否必须经过策略层。

### 4.3 第三层：策略路由

策略关注“即使任务上应该做，当前主体是否允许做”。输入至少包括 actor、role、tenant、resource、action、risk、environment。输出常用四值：`allow / deny / require_approval / allow_with_constraints`。

策略拒绝不能被模型“说服”，也不能因为模型置信度高而绕过。策略层需要默认拒绝、版本化规则和审计记录。

### 4.4 第四层：编排与故障路由

运行时根据错误类型决定：

- 参数错误：回到模型修参，但限制次数；
- 可重试瞬时错误：指数退避并尊重幂等；
- Provider 限流：切换队列或 Provider；
- 业务证据不足：澄清、降级答案或拒答；
- 高风险不确定：HITL；
- 不可恢复：终止并保留完整 trace。

不要用一个 `except Exception: fallback()` 吞掉所有故障。否则认证失败也可能错误切到另一条更宽松的路径。

## 5. 关键设计：冲突、置信度、默认分支与质量指标

### 5.1 冲突处理

多规则系统应显式定义：

- **priority**：数字越大是否越优先，必须统一；
- **specificity**：资源、事件、条件更具体者优先；
- **stable tie-breaker**：同优先级时按规则 ID 或配置顺序，避免随机行为；
- **terminal**：命中后是否停止继续匹配；
- **composition**：多个 action 是合并、覆盖还是互斥。

默认规则应最后求值，并能从测试中证明任何有效输入都有去向。默认 `allow` 通常不适合高风险动作。

### 5.2 confidence 不是天然概率

模型输出 `0.92` 不代表现实中有 92% 正确率。可以在标注集上做 reliability diagram、ECE、Brier score 或按业务桶统计真实命中率。阈值也不能只追求总体 accuracy：漏拦高风险操作与多问一次澄清的损失不同。

三段式阈值示例：

- \(c \ge t_{auto}\)：进入候选路线，仍经过策略校验；
- \(t_{clarify} \le c < t_{auto}\)：请求澄清；
- \(c < t_{clarify}\)：拒绝或走只读安全 fallback。

### 5.3 路由后悔成本

仅统计分类准确率会掩盖业务差异。可定义：

\[
L = C_{wrong}(y, \hat y) + \lambda_1 C_{latency} + \lambda_2 C_{token} + \lambda_3 C_{risk}
\]

`report -> chat` 可能只是体验损失，`read -> destructive_write` 则是安全事故。成本矩阵应由业务、安全和平台共同确定。

### 5.4 可观测性

至少记录：输入特征摘要、候选集合、规则命中、模型版本、结构化输出、校准后置信度、策略结果、最终路线、fallback 原因、延迟和结果标签。敏感原文应脱敏或使用受控存储，不要为了可观测性制造泄露。

## 6. ESGQA 证据、调用链与事实边界

证据：`P1-ESG-ROUTING-001`、`P1-ESG-STRUCTURED-OUTPUT-001`。

冻结源码中的控制流可概括为：

```text
START
  -> memory
  -> intent_parser
  -> route_intent(state.intent.intent)
       rate_query    -> factor_extractor -> check_factor
       report_query  -> ...
       chat          -> ...
  -> 若干 check_* 条件边
  -> END 或后续领域节点
```

`build_cbam_graph()` 创建 `StateGraph(CBAMState)`、注册节点和边；`route_intent` 读取结构化意图并映射到分支；`check_factor` 等函数根据错误状态选择继续或错误路径。Pydantic 模型为意图、因子、利率、RAG 评估结果提供结构化形状。

可以严格陈述：

- **[已实现]** 显式状态图和条件边；
- **[已实现]** 意图与若干节点后错误状态驱动分支；
- **[已实现]** 结构化模型输出降低自由文本解析不确定性；
- **[可演示]** 不同问题进入不同领域链，以及错误状态如何改道。

不能扩大为：

- **[未实现]** 通用、可热更新、带版本与优先级的 Router 服务；
- **[未实现]** 置信度校准、成本矩阵和离线路由评测；
- **[未实现]** Router 统一决定任意 Tool、Provider 或 Agent；
- **[未实现]** 独立 Policy Engine 或高风险 HITL 门。

改进建议：让意图解析输出 `route + reason_code + raw_confidence + evidence`，在图外增加确定性候选过滤和策略否决；用历史标注请求建立混淆矩阵，重点测 `rate_query/report_query/chat` 的高代价误路由，而不是只看总体正确率。

## 7. agentHub 证据、调用链与事实边界

证据：`P1-HUB-ROUTING-001`、`P1-HUB-PLANNER-001`。

`EventRoutingRules.ts` 定义事件条件、路由动作和优先级，规则匹配后形成目标 Agent/动作选择；这属于事件控制面的显式规则路由。`PlannerAgent.ts` 则生成计划、校验并在有界次数内重试；`ManagerLoop.ts` 根据执行结果选择继续、重规划或终止，体现故障与进度决策。

可以严格陈述：

- **[已实现]** 有类型的事件路由规则与优先级；
- **[已实现]** 规则条件匹配及动作选择；
- **[已实现]** Planner 的校验和有限重试；
- **[已实现]** Manager 的 continue/replan/abort 决策。

必须保留的边界：

- **[需核验/改进]** 同优先级多规则冲突的稳定性、组合语义与覆盖测试；
- **[未见完整实现]** 模型路由 confidence 校准与成本敏感评测；
- **[未见完整实现]** 独立策略语言、决策版本和统一审计解释；
- **[改进建议]** 将 `matchedRuleId`、候选规则、最终 action 和策略覆盖原因写入 trace。

两个项目不要混讲：ESGQA 是领域状态图中的条件路由；agentHub 更偏运行时事件路由与 Planner/Manager 决策。它们可以共同证明你理解多层控制流，但不能互相“借实现”。

## 8. 教学代码示例：可审计分层 Router

> [!warning] 教学代码边界
> 以下代码仅用于解释接口和控制顺序，未在两个项目中运行，也不表示项目已经实现此抽象。

```python
from dataclasses import dataclass
from typing import Literal

Route = Literal["answer", "report", "clarify", "reject"]

@dataclass(frozen=True)
class RouteDecision:
    route: Route
    reason_code: str
    confidence: float
    matched_rule: str | None
    policy_effect: Literal["allow", "deny", "approval"]
    router_version: str

def route(req, model_router, policy) -> RouteDecision:
    # 硬规则只处理确定且高优先级的事实。
    if not req.authenticated:
        return RouteDecision("reject", "AUTH_REQUIRED", 1.0,
                             "auth-required", "deny", "2026-07-22")

    candidate = model_router.classify(
        text=req.text,
        choices=["answer", "report"],
    )
    if candidate.route not in {"answer", "report"}:
        return RouteDecision("clarify", "INVALID_CANDIDATE", 0.0,
                             None, "allow", "2026-07-22")

    if candidate.calibrated_confidence < 0.72:
        return RouteDecision("clarify", "LOW_CONFIDENCE",
                             candidate.calibrated_confidence,
                             None, "allow", "2026-07-22")

    effect = policy.evaluate(req.actor, candidate.route)
    if effect != "allow":
        final = "reject" if effect == "deny" else "clarify"
        return RouteDecision(final, "POLICY_GATE",
                             candidate.calibrated_confidence,
                             None, effect, "2026-07-22")

    return RouteDecision(candidate.route, "MODEL_ROUTE",
                         candidate.calibrated_confidence,
                         None, "allow", "2026-07-22")
```

面试时要指出：示例省略了模型异常、Schema 验证、幂等、HITL 状态机、脱敏 trace 和真实校准流程；这些不是“几行代码就完成”的能力。

## 9. 文字 Demo：白板演示与项目素材映射

### Demo A：ESGQA 业务意图路由

输入一：“某公司 2023 年碳排放指标是多少？”

1. `memory` 组装上下文；
2. `intent_parser` 产生结构化意图；
3. `route_intent` 选择指标/报告相关领域链；
4. 后续节点若设置错误状态，`check_*` 选择错误分支；
5. 展示图定义而非声称模型自由调用任意工具。

输入二：“你好，解释一下 ESG 是什么。”对比其进入 chat/通用回答分支。演示重点是**相同入口、不同显式边**。

### Demo B：agentHub 事件规则冲突

构造两个都能匹配同一事件的规则：一个路由到专业 Agent，一个路由到默认 Agent。白板列出 priority、条件、动作，追问：同优先级怎么办？规则未命中怎么办？目标 Agent 不可用怎么办？当前源码能证明有优先级匹配，但生产答案必须补充稳定 tie-breaker、健康检查和 trace。

### Demo C：低置信度不是“随便选一个”

给出 `0.51 report / 0.49 answer`。合理路径是澄清或低风险 fallback，而不是因为 0.51 最大就自动生成高成本报告。再把动作改为删除文件，说明阈值必须随风险变化。

## 10. 常见错误、反例与排障路径

1. **错误：把 Router 当分类器。** 排障：列出决策是否包含策略、故障和运行时能力。
2. **错误：直接相信模型 confidence。** 排障：查标注集、校准方法、阈值依据和时间漂移。
3. **错误：默认分支总是 allow。** 排障：对未知事件、高风险动作做负向测试。
4. **错误：规则无版本。** 排障：无法从历史 trace 重放当时决策时，审计链不完整。
5. **错误：捕获所有异常后 fallback。** 排障：按认证、参数、限流、瞬时、业务、未知错误分类。
6. **错误：路由环没有步数上限。** 排障：记录 route history，检测重复状态和最大 hop。
7. **错误：候选能力不可用仍被选中。** 排障：路由前做 capability/health snapshot，并记录版本。
8. **错误：规则命中却无解释。** 排障：输出 matched rule、条件摘要和最终覆盖关系。

排障顺序建议：复现输入与版本 -> 查看候选与规则 -> 验证模型结构化输出 -> 查看策略覆盖 -> 查看运行时故障路由 -> 对比最终业务结果。不要先改 Prompt。

## 11. 面试追问树、回答骨架与危险答案

### 主问题：设计一个生产 Router

30 秒骨架：先硬规则与策略否决，再模型处理语义长尾；输出结构化决策；低置信度澄清；所有结果带版本和 trace。

2 分钟展开：

1. 定义候选与 `RouteDecision`；
2. 规则、模型、策略、故障四层；
3. confidence 校准和风险分层阈值；
4. fallback/HITL/终止；
5. 离线混淆矩阵、成本矩阵与线上监控；
6. 用 ESGQA 条件边和 agentHub 优先级规则举证并说明边界。

追问：为什么不用一个大模型全做？

答：高风险硬约束需要确定性、可审计和默认拒绝；模型适合补语义长尾，但不应同时拥有策略解释与执行授权。

追问：如何测？

答：构造按业务代价分层的标注集，测 per-route precision/recall、拒绝/澄清率、校准误差、路由后悔成本、P95 延迟；再做规则冲突、未知输入、Provider 不可用和策略拒绝测试。

危险答案：

- “temperature=0 就完全确定。”
- “输出是 Pydantic，所以业务一定正确。”
- “取最高概率即可。”
- “异常统一重试三次。”
- “两个项目都有完整通用 Router 平台。”

## 12. 大厂级设计检查清单与评审口径

- [ ] 候选路由有 owner、输入前置条件和风险等级。
- [ ] 决策结果有 route、reason、版本、trace、policy effect。
- [ ] 高优先级拒绝规则不受模型输出覆盖。
- [ ] 多规则命中有稳定、可测试的冲突语义。
- [ ] 默认分支和未知输入行为明确。
- [ ] 模型输出经过 Schema、枚举、业务与策略验证。
- [ ] confidence 在自有数据上校准，阈值绑定业务损失。
- [ ] fallback 不会扩大权限或绕过认证。
- [ ] 重试只针对可重试错误，并考虑幂等。
- [ ] 有最大 hop、循环检测、deadline 和取消传播。
- [ ] trace 可重放决策，同时保护敏感数据。
- [ ] 离线与线上指标能定位到 route/rule/model/version。
- [ ] 项目表述区分 `[已实现]`、`[可演示]`、`[改进建议]`、`[未实现]`。

## 13. 复习卡、口述任务与自我检查

### 必背卡

- Router 四层：规则、模型、策略、故障/编排。
- confidence：模型分数不等于校准概率。
- fallback：不能绕过权限，不能吞掉错误分类。
- 评测：业务成本矩阵优先于单一 accuracy。
- ESGQA：领域状态图条件边；agentHub：事件优先级规则与管理决策。

### 口述任务

1. 用 90 秒解释 Router 和分类器的区别。
2. 用白板画 `request -> hard rules -> model candidate -> policy -> execute`。
3. 分别给出一个应澄清、应拒绝、应 HITL、应自动 fallback 的例子。
4. 在不看笔记时说清两个项目各自能证明什么、不能证明什么。

### 自我检查答案

问：为什么先做候选过滤？答：避免模型选择当前不可用、无权限或不满足前置条件的路线，缩小不确定空间。

问：为什么高风险阈值更高仍不够？答：高置信度也可能错，仍需确定性策略和必要的人审。

问：规则版本为什么重要？答：历史行为取决于当时规则，审计和重放必须知道当时版本。
