# Lesson 1.6：Context Engineering、记忆与压缩

## 1. 课程定位、前置依赖与后续伏笔

Agent 的质量不只取决于模型，还取决于每一步“让模型看见什么、按什么顺序、占多少预算、为何可信”。Context Engineering 是运行时的数据选择与编排问题，不是把数据库内容全部拼进 Prompt。

本课连接四个系统面：

- Agent Loop 每一步会产生新观察并消耗窗口；
- Tool/检索结果可能体积大、包含不可信指令或敏感数据；
- Provider 返回实际 Usage，且窗口、计数方式和缓存语义不同；
- Security 要治理记忆污染、跨用户泄露和外部内容注入。

后续课程会将 Prompt Injection、数据边界和日志治理落实为安全要求。更深入的 RAG 召回、重排和评测可在后续 Phase 展开，本课只建立面试所需的系统框架。

## 2. 学习目标、应答问题与课前诊断

完成后应能回答：

## 3. 一句话模型、四类上下文与预算方程

> **一句话模型：Context Engineering 是在有限模型窗口内，按相关性、重要性、时效性、可信度和成本选择信息，并维护可恢复、可评估的压缩与记忆生命周期。**

先区分四类对象：

1. **Local runtime context**：数据库连接、logger、tenant、trace、权限对象等，仅代码可见，不自动进入模型。
2. **Model-visible context**：system/developer/user 消息、历史、工具定义、检索结果和工具结果，实际消耗窗口。
3. **External memory**：数据库、向量库、ContextBus 等可持久或跨步骤的数据源，需要选择后才进入模型。
4. **Execution state**：计划、待处理工具、重试计数、压缩状态等控制数据，部分可只留在运行时。

预算方程：

\[
B_{window} \ge B_{fixed} + B_{history} + B_{memory} + B_{retrieval} + B_{tools} + B_{input} + B_{outputReserve} + B_{safety}
\]

`B_fixed` 包括系统/开发者指令和输出契约；`B_tools` 包括工具 Schema；`B_outputReserve` 必须在调用前预留；`B_safety` 是估算误差和 Provider 计数差异的余量。实际计数优先使用 Provider tokenizer/Usage，字符估算只能做预筛。

## 4. 上下文选择：优先级、位置和最便宜的减量顺序

### 4.1 信息价值函数

可用一个工程启发式排序，而不是声称存在万能公式：

\[
score(i) = w_r relevance + w_p priority + w_f freshness + w_a authority
           - w_c tokenCost - w_u uncertainty - w_s securityRisk
\]

强制固定的系统约束和当前用户请求不应与普通历史使用同一排序竞争；它们是 protected/pinned 区域。检索结果还需考虑来源、多样性、重复和冲突。

### 4.2 “最便宜先做”

建议的减量顺序：

1. 删除重复、模板噪声、空字段和无意义日志；
2. 工具只返回模型需要的字段，而非原始全集；
3. 去除已完成且无后续价值的中间事件；
4. 按权限与相关性检索，而非全量拼接；
5. 对长文档先切片、重排、去重；
6. 对历史进行结构化摘要；
7. 最后才压缩仍然关键的信息。

“直接总结所有历史”并不便宜：它增加一次模型调用，还可能把错误事实固化为长期记忆。

### 4.3 位置效应

`Lost in the Middle` 研究提醒：窗口容量大不代表模型对所有位置同等利用。工程上可把最稳定的指令置于受保护开头，把当前任务、关键证据与输出要求靠近决策点；但不要依靠位置替代权限控制或事实验证。

### 4.4 Tool Schema 与结果也是预算消费者

几十个冗长工具描述会挤占历史和输出预算。可按阶段只暴露候选工具、缩短描述但保留关键约束、把巨型 ToolResult 存入外部对象并回传句柄与摘要。句柄读取仍须鉴权和租户隔离。

## 5. 记忆与压缩：写入治理、状态机和质量评测

### 5.1 记忆不是聊天记录别名

建议把记忆分为：

- episodic：发生过的事件，有时间与来源；
- semantic：稳定事实或用户偏好，可修订；
- procedural：流程、约束和经验规则；
- working：当前任务临时状态；
- artifact/index：大对象引用及其元数据。

写入前回答：谁说的？是否验证？属于哪个用户/租户？何时失效？能否编辑删除？是否敏感？与旧记忆冲突怎么办？

### 5.2 摘要的损失

摘要可能丢掉否定词、数字、例外条件、未完成任务、来源与不确定性。高价值事实应结构化保留：`claim/source/time/confidence/status`。摘要应引用原始 span 或事件 ID，允许追溯，而不是覆盖原文成为唯一真相。

### 5.3 压缩状态机

可靠流程包含：

```text
ACTIVE
  -> threshold crossed
COMPRESSION_PENDING
  -> 暂存新用户输入，冻结旧会话继续扩张
COMPRESSING
  -> 生成并验证摘要/保留项
RESTARTING
  -> 新 Provider 会话加载摘要 + pinned state + pending input
ACTIVE | FAILED
```

压缩期间到达的 Prompt 不能同时进入旧上下文，否则会出现竞态、重复处理或丢消息。

### 5.4 阈值与迟滞

单阈值 70% 可工作，但容易在边界反复触发。可设置触发阈值 75%、恢复目标 45%，并加入冷却时间/最小新增 Token。阈值必须由任务长度、输出 reserve、压缩延迟和质量数据校准，不存在全局最佳数字。

### 5.5 质量评测

建立“压缩前可答、压缩后仍应可答”的探针集：用户硬约束、未完成计划、关键数字、错误原因、来源、权限状态。测 factual retention、constraint retention、task success、hallucinated additions、compression ratio、latency/cost。摘要为空、验证失败时应保守回退，不把“Summary unavailable”当有效上下文继续高风险任务。

## 6. ESGQA 证据、调用链与事实边界

证据：`P1-ESG-MEMORY-001`、`P1-ESG-RETRIEVAL-001`。

### 6.1 记忆组装

`node_memory()` 的冻结调用链：

```text
SQLite 最近 4 条对话
  + SQLite 长期画像
  + FAISS 相似搜索兜底/补充
  -> memory_context
  -> 意图解析节点使用
```

SQLite 异常会降级为空字符串；没有数据库结果时使用 FAISS 文本。可以说项目区分近期对话、长期画像与向量记忆，并在意图识别前组装。

边界：

- **[未实现/未见证据]** 显式 Token 预算与输出预留；
- **[未见证据]** TTL、用户编辑/删除、租户级记忆治理；
- **[风险]** 固定 `user_001` 不足以证明多用户隔离；
- **[需改进]** FAISS 搜索异常在该函数内没有同样的降级保护；
- **[需改进]** 记忆内容的来源、置信度和冲突处理。

### 6.2 检索上下文

`ESGHybridRetriever.search()`：Redis 缓存 -> BM25 与 FAISS 召回 -> 0.4/0.6 Ensemble -> 元数据和启发式过滤 -> BGE Reranker -> 业务规则提权 -> 缓存写回。混合模式扩大候选后精排，体现了多阶段选择。

严格边界：源码证明流程，不证明当前线上召回率、准确率或缓存收益；`allow_dangerous_deserialization=True` 要求索引文件可信；检索结果进入 Prompt 前仍需预算、来源和注入治理。

## 7. agentHub 证据、调用链与事实边界

证据：`P1-HUB-CONTEXT-001`、`P1-HUB-COMPRESSION-001`。

### 7.1 ContextBus

`ContextBus.ts` 将上下文存成带类型、标签、状态、版本和作者的条目。查询时使用类型基础权重、7 天衰减和引用奖励，`getProjectDigest(maxTokens)` 在预算内构建 digest，并有归档/GC。

可以说：

- **[已实现]** 结构化条目与版本化写入；
- **[已实现]** 类型/标签/状态查询；
- **[已实现]** 预算选择、启发式权重、归档和 GC；
- **[限制]** 字符截断与权重是启发式，没有当前证据证明最优或信息保真；
- **[限制]** 该类本身不能单独证明跨重启持久化全链路。

### 7.2 70% 压缩流程

`AgentRuntime.ts` 根据 Usage 计算 context 百分比，超过 70% 设置压缩状态；下一个 Prompt 暂存；向 Provider 发送压缩请求；收到摘要后重启 Provider，并用 summary + pending prompt 恢复。

可以说它实现了基于 Usage 的压缩状态机，并处理压缩期间新 Prompt 的串行化。不能说 70% 是最佳阈值，也不能说摘要不丢信息。源码中的摘要为空 fallback 可保持流程继续，但生产上还应增加摘要质量门、pinned facts、迟滞和失败恢复。

## 8. 教学代码示例：预算分配与选择器

> [!warning] 教学代码边界
> 以下代码仅用于解释预算算法，未在两个项目中运行。字符/Token 估算必须替换为目标 Provider 的可靠 tokenizer 或实际 Usage。

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class Item:
    key: str
    tokens: int
    score: float
    pinned: bool = False

def select_context(items: list[Item], window: int,
                   fixed: int, user_input: int,
                   output_reserve: int, safety: int) -> list[Item]:
    available = window - fixed - user_input - output_reserve - safety
    if available < 0:
        raise ValueError("FIXED_CONTEXT_EXCEEDS_WINDOW")

    selected: list[Item] = []
    pinned = [x for x in items if x.pinned]
    optional = [x for x in items if not x.pinned]

    for item in pinned:
        if item.tokens > available:
            raise ValueError(f"PINNED_ITEM_TOO_LARGE:{item.key}")
        selected.append(item)
        available -= item.tokens

    # 简化的价值密度启发式；生产实现还需去重、多样性与信任边界。
    for item in sorted(optional, key=lambda x: x.score / max(x.tokens, 1), reverse=True):
        if item.tokens <= available:
            selected.append(item)
            available -= item.tokens
    return selected
```

关键设计是固定区超限时明确失败，而不是悄悄截断系统约束。可选区算法也不是“最优证明”，只是可解释起点。

## 9. 手算预算与文字 Demo

### 9.1 手算预算

假设窗口 32,000 tokens：

| 消费项 | 预算 |
|---|---:|
| system/developer 与输出 Schema | 3,000 |
| 当前用户输入 | 1,500 |
| Tool Schema | 2,500 |
| 输出预留 | 6,000 |
| 安全余量 | 2,000 |
| 可分配给历史/记忆/检索 | 17,000 |

若历史 8,000、记忆候选 6,000、检索候选 12,000，总计 26,000，不能全塞。一个可解释方案：历史保留 5,000（当前回合+未完成计划）、结构化记忆 3,000、重排后的检索 9,000，总计 17,000。若输出需求提升到 10,000，应先收缩可选上下文，而不是赌模型超窗。

### 9.2 Demo A：ESGQA 记忆来源

展示最近 4 条、长期画像和 FAISS 文本如何组合。追问同一事实冲突时谁优先、固定 user ID 如何多用户化、异常后降级是否可观测。该 Demo 强调“来源分层已经有，生命周期治理还没有”。

### 9.3 Demo B：ESGQA 多阶段检索

用同一查询展示 BM25 词面命中、FAISS 语义命中、Reranker 重排和业务规则提权的职责。不要给出未测量的准确率；用“代码路径证明组件存在”表述。

### 9.4 Demo C：agentHub ContextBus

准备 convention、decision、known-issue 和普通条目，按预算展示类型权重、时间衰减、引用奖励和截断。追问字符截断可能切断 JSON/代码/否定句，提出结构化边界截断与 token-aware 改造。

### 9.5 Demo D：70% 压缩竞态

在 usage 超过 70% 后立刻到达新 Prompt：它被暂存，摘要完成并重启 Provider 后再恢复。演示状态机价值，再指出缺少迟滞和摘要保真门。

## 10. 常见错误、反例与排障路径

1. **把 local context 自动当成模型记忆。** 只有序列化到模型输入的内容才可见。
2. **先填满输入再考虑输出。** 会导致截断或无法生成完整结构化结果。
3. **只保留最近 N 条。** 可能丢掉旧但仍有效的硬约束。
4. **摘要覆盖原始记录。** 失去追溯和纠错能力。
5. **记忆无租户键。** 会产生跨用户泄露风险。
6. **检索 top-k 固定。** 文档长度、重复和查询难度变化时预算失控。
7. **字符数等于 token 数。** 多语言、代码和 Provider tokenizer 差异明显。
8. **压缩触发后仍接收旧会话输入。** 产生竞态和重复执行。
9. **把工具原始日志完整回注。** 污染上下文并扩大注入面。
10. **大窗口等于不用检索。** 成本、位置利用、更新和权限问题仍存在。

排障顺序：获取真实 Usage -> 分解每类消费者 -> 检查固定/输出 reserve -> 对比选择前后条目 ID -> 检查截断与摘要 -> 重放压缩状态 -> 用探针验证关键约束 -> 检查租户和来源。

## 11. 面试追问树、回答骨架与危险答案

### 主问题：如何管理 Agent 长上下文？

30 秒回答：先分本地上下文、模型输入、外部记忆和执行状态；按固定区、输出预留、安全余量计算预算；历史摘要、结构化记忆与检索按价值选择；Usage 触发有状态压缩；用保真探针评测。

2 分钟展开：

1. 列出窗口消费者和预算公式；
2. pinned 与 optional 分区；
3. 去重、工具结果裁剪、检索、摘要的减量顺序；
4. 记忆来源/TTL/冲突/删除/租户治理；
5. 压缩状态机、迟滞和恢复；
6. token、质量、延迟、成本、安全指标；
7. 用 ESGQA 与 agentHub 证据说明两类实现。

追问：为什么不用 1M 长窗口？

答：容量不等于均匀利用；成本、延迟、敏感数据、工具 Schema、更新、权限和位置效应仍要管理。窗口更大只是预算上限变化，不取消选择问题。

追问：如何验证摘要没丢关键信息？

答：保留来源 ID，定义 constraint/fact/task 探针，压缩前后回答对比；关键字段结构化 pin；失败时回退或人审，不把 ROUGE 当唯一业务指标。

危险答案：

- “最近十轮就够了。”
- “向量库就是长期记忆。”
- “超过 70% 总结一下，绝不会丢信息。”
- “上下文越长效果越好。”
- “ESGQA 已有完整企业级记忆治理。”

## 13. 复习卡、口述任务与自我检查

### 必背卡

- 四层：本地上下文、模型可见上下文、外部记忆、执行状态。
- 预算：窗口减固定、输入、工具、输出 reserve、安全余量。
- 减量：去噪 -> 工具裁剪 -> 检索/去重 -> 结构化摘要 -> 压缩。
- 压缩：pending、compressing、restart、resume，需要迟滞与保真评测。
- 项目：ESGQA 有近期/画像/FAISS 和混合检索；agentHub 有 ContextBus 与 70% 压缩状态机。

### 口述任务

1. 不看笔记写出预算方程并手算 32K 示例。
2. 用 2 分钟解释“记忆不是向量库”。
3. 画压缩状态机，说明压缩期间新 Prompt 去哪。
4. 分别指出两个项目的一个亮点、一个风险、一个优化建议。

### 自我检查答案

问：为什么未完成计划要 pin？答：它决定下一步和终止条件，被普通相关性排序淘汰会导致 Agent 偏航。

问：为什么工具结果应有句柄？答：大对象留在受控存储，模型只看必要摘要；但句柄读取仍要鉴权和过期治理。

问：为什么摘要要保留来源？答：摘要可能误写，来源支持核验、纠错和审计。
