# Lesson 1.2：Agent Loop 核心模式

## 1. 课程定位、前置依赖与后续伏笔

上一课给出了 `state -> action -> observation -> state'`。本课回答工程上最容易失控的问题：**由谁推进这条闭环，状态如何转移，失败如何停止，轨迹如何复现？**

前置依赖：理解 Agent 是系统而非模型，能区分运行时状态和模型上下文。后续 Tool-use 会填充 Loop 中的行动协议，Router 会把“下一步”从布尔判断升级为显式决策，Provider 会把模型差异隔离到边界。

重点不是背四个名词，而是能从任务约束选择：

- 直接 while-loop；
- 显式 StateGraph；
- 事件驱动 Runtime；
- ReAct、Plan-Execute、ReWOO 或混合控制策略。

> [!important] 面试必修专题
> 本课负责 Loop 主线，[[../general-agent-architecture-patterns|通用 Agent 架构模式专题]]负责完整展开 ReAct、Planning、ReWOO、Reflexion、Self-Refine、CRITIC、ToT、GoT、LATS、LLMCompiler、Multi-Agent 与 Hybrid。项目是否采用不影响这些知识的课程地位；项目映射必须另行标注已实现、局部相似或未实现。

## 2. 学习目标、应答问题与课前诊断

完成后应能回答：

- while-loop、StateGraph、事件驱动 Runtime 的控制权分别在哪里？
- ReAct 为什么能根据观察修正，代价是什么？
- Plan-Execute 和 ReWOO 为什么能减少部分模型调用，又会牺牲什么？
- max_steps、deadline、budget、no_progress 有何区别？
- 重试为什么不是简单 `except: continue`？
- 并行 ToolCall 如何保持结果关联和确定性？
- ESGQA 与 agentHub 分别属于哪类控制流？

课前诊断：如果你写过 `while True`，但只有模型输出 `FINAL` 才 break，没有最大步数、异常分类和重复检测，那么你写的是“能跑的循环”，不是“受控 Agent Runtime”。

## 3. 一句话模型、问题动机、非目标与适用边界

> **一句话模型：Agent Loop 是由运行时掌控的状态转移协议，它把模型提议、工具观察、策略检查和终止条件串成可界定、可观测、可恢复的执行轨迹。**

问题动机：复杂任务的下一步依赖上一动作的真实结果。若把所有步骤预先写死，无法适应未知结果；若把全部控制权交给模型，又无法保证安全、停止和审计。Loop 的职责是在动态性与确定性之间建立边界。

非目标：

- 不把模型隐藏推理过程保存为系统真相。
- 不假设每轮都必须调用 LLM。
- 不默认失败就重试。
- 不把消息队列、图框架或异步函数本身等同于 Agent。

适用边界：简单两步工具调用可以用普通函数；需要条件分支和恢复但路径可枚举时用图；跨请求、长生命周期、并发事件和外部进程时需要 Runtime/队列。

## 4. A 层完整推导：控制流、状态转移与四类模式

### 4.1 Loop 的状态机定义

令运行状态：

\[
s_t = (h_t, b_t, p_t, e_t, k_t)
\]

- `h_t`：模型可见历史或上下文引用；
- `b_t`：Token、步骤、时间、费用与工具预算；
- `p_t`：权限和审批状态；
- `e_t`：最近错误与重试计数；
- `k_t`：进展指纹、检查点和幂等键。

每轮不是“再问一次模型”，而是执行转移函数：

\[
s_{t+1} = F(s_t, Route(Policy(Context(s_t))))
\]

其中 `Route` 可以直接完成、执行工具、等待人工、降级或终止。

### 4.2 模式一：直接 while-loop

```text
while limits allow:
    response = provider.complete(context, tool_schemas)
    if response.final:
        return verify(response.final)
    for call in response.tool_calls:
        result = execute(call)
        append(result)
return terminate(reason)
```

优点：控制流直观、依赖少、适合单 Agent 和少量工具。缺点：分支增多后异常、重试、人工中断、恢复和并发会挤在一个函数里。

适合：教学、原型、短请求；不适合：跨小时执行、多参与者、复杂检查点和事件订阅。

### 4.3 模式二：StateGraph

图把状态转移显式化：节点负责局部计算，边负责下一节点。固定边表达必经阶段，条件边表达决策，循环边表达重试或迭代，END 表达终止。

优点：路径可视化、节点可独立测试、业务分支和 fallback 清楚、易做检查点。缺点：动态任务可能产生“节点爆炸”；共享状态字段会形成隐式耦合；子图、并行和回放需要框架语义支持。

关键判断：图并不天然比 while-loop 更 Agentic，它只是把控制流变得显式。

### 4.4 模式三：事件驱动 Runtime

事件驱动系统不是同步地“调用下一函数”，而是组件产生事件：`provider_started`、`tool_use`、`permission_request`、`usage`、`done`、`error`。Runtime 根据事件更新状态、广播、排队或清理资源。

优点：适合长生命周期、流式输出、外部进程、并发用户和多 Agent；组件解耦。缺点：时序更难理解；重复、乱序、丢失和至少一次投递会引入一致性问题；必须设计 event_id、causation_id、幂等与重放。

### 4.5 模式四：确定性 Workflow 包裹局部 Agent

这是生产中常见且稳健的混合：

```text
固定入口校验
-> 固定数据准备
-> 局部 Agent 调研/选择工具
-> 固定结果验证
-> 人工审批
-> 固定提交/发布
```

它把高风险副作用留在确定性阶段，把开放搜索空间限制在局部。

### 4.5A 架构层次：不要把策略、控制流和 Runtime 混在一起

ReAct、ReWOO、ToT 属于决策/搜索策略；while-loop、StateGraph、DAG 是控制流；Provider 事件、队列、取消和终止属于 Runtime；Tool/Policy/Sandbox 属于能力与治理。一个生产系统可以用 StateGraph 承载局部 ReAct，也可以由事件 Runtime 执行 Planner 生成的 DAG。完整比较见 [[../general-agent-architecture-patterns]]。

项目归属必须保守：ESGQA 的显式图不等于 GoT，agentHub 的事件循环不等于 ReAct，有 Planner 也不自动等于 ReWOO。

### 4.6 ReAct

[ReAct](https://arxiv.org/abs/2210.03629) 的核心是推理与行动/观察交错：模型根据最新观察调整下一步。工程价值是适应性和故障修正；工程代价是每一步都可能增加模型调用、上下文和错误传播。

不要把公开 Chain-of-Thought 当接口。生产实现只需保存**结构化决策理由摘要、动作、观察和终止原因**，而不是要求或记录模型私有推理全文。

### 4.7 Plan-Execute

先由 Planner 生成子任务或 DAG，再由 Executor 执行。它减少每一步重新规划的成本，也让任务可分派、并行和审计。风险是早期计划建立在不完整信息上；执行观察变化后必须支持 replan，否则会沿错误计划继续。

### 4.8 ReWOO

[ReWOO](https://arxiv.org/abs/2305.18323) 将 reasoning/planning 与工具观察解耦，典型角色是 Planner、Worker、Solver。计划中用变量引用未来观察，Worker 执行工具，Solver 汇总。它可以减少模型在每个工具结果后的重复推理，但实时适应性弱于 ReAct。

### 4.9 选型结论

| 模式 | 适应未知观察 | 模型调用 | 可审计 | 并行潜力 | 主要风险 |
|---|---:|---:|---:|---:|---|
| ReAct | 高 | 高 | 中 | 低到中 | 循环、成本、错误传播 |
| Plan-Execute | 中 | 中 | 高 | 高 | 初始计划失真 |
| ReWOO | 中低 | 较低 | 高 | 高 | 观察后难重规划 |
| 固定 Workflow | 低 | 可控 | 很高 | 由图决定 | 对未知情况僵硬 |

不存在全局最优。先问任务是否需要观察后重新决策，再问并行、成本和审计要求。

## 5. 不变量、停止条件、反模式与边界条件

### 5.1 五类独立停止条件

1. **目标完成**：结果通过外部验证，不是模型自报完成。
2. **最大步数**：控制最坏循环次数。
3. **截止时间**：控制墙上时间；工具卡住时步数不变也会超时。
4. **预算耗尽**：Token、费用、外部 API、工具次数。
5. **无进展**：状态或行动指纹重复，避免在限制内做无意义循环。

此外还有 Policy Abort、人工取消、不可恢复错误。多个条件可能同时满足，运行时应定义优先级并记录首要原因与附加原因。

### 5.2 进展指纹

可以对以下稳定 JSON 求哈希：动作类型、工具名、规范化参数、结果状态、关键状态版本。连续相同指纹不一定代表无进展，例如分页轮询；因此需结合任务语义、结果变化和等待策略，而不是“相同两次就杀死”。

### 5.3 重试语义

| 失败 | 默认动作 | 原因 |
|---|---|---|
| Provider 429/临时网络 | 有上限退避 | 通常瞬时 |
| Provider 认证失败 | 立即终止/切换 | 重试同配置无意义 |
| 参数 Schema 错误 | 把结构化错误返回模型一次 | 模型可能修正 |
| 权限拒绝 | 等待审批或终止 | 重试不会改变权限 |
| 只读 Tool 超时 | 可在幂等前提下重试 | 副作用低 |
| 写 Tool 超时 | 先查询结果/幂等键 | 操作可能已发生 |

### 5.4 并发不变量

- 每个 ToolCall 有稳定 call_id，ToolResult 必须关联同一 ID。
- 并行只适用于无依赖或依赖显式的任务。
- 汇总结果要采用稳定排序，避免同输入不同轨迹。
- 共享状态使用版本或 compare-and-swap，防止最后写入覆盖。
- 取消要向子任务传播，并记录已发生副作用。

### 5.5 反模式

- `while True` + 字符串包含 “FINAL”。
- catch 所有异常后继续。
- 把工具失败文本拼给模型，但不设置错误类型。
- 每次重试都重新生成随机幂等键。
- busy 时直接并发调用同一有状态 Provider。
- 把事件日志当事实存储，却没有幂等消费。

## 6. ESGQA 证据、状态流与事实边界

证据：`P1-ESG-STATEGRAPH-001`、`P1-ESG-ROUTING-001`。

ESGQA 的主链在 `agents.py:906-966`：节点注册、固定边、条件边、END 都显式。`route_intent()` 负责一级业务分支，五个 `check_*()` 将各阶段的 `error_msg` 转入 fallback。

这能证明：

- 状态由 `CBAMState` 跨节点传递；
- 路径可被源码枚举；
- 成功和失败都有终点；
- 不同任务意图进入不同节点链。

不能证明：

- 通用 ReAct while-loop；
- 模型动态创建计划或任意选择工具；
- max_steps、deadline、Token budget 和 no-progress 统一治理。

面试评价：对于 CBAM 合规核算，显式图是优点，因为每个业务阶段可审计。真正的改造重点不是“换成更自由的 Agent”，而是补齐统一终止元数据、节点超时、检查点、幂等和运行指标。

## 7. agentHub 证据、状态流与事实边界

证据：`P1-HUB-RUNTIME-001`、`P1-HUB-PLANNER-001`。

### 7.1 Runtime

`AgentRuntime.ts` 在 busy 时排队、idle 时发送 prompt；接收 Provider 统一事件；处理权限请求、上下文压缩、下一队列和 idle timeout。它是事件驱动生命周期控制，不是教材式单 while-loop。

### 7.2 Planner/Manager

`PlannerAgent.ts` 生成 JSON 计划后调用 `extractAndValidate`，解析失败有限重试；`ManagerLoop.ts` 在任务失败时收集错误、输出、上游结果和文件树，让 Main Agent 选择 `continue/replan/abort`，非法或异常决策回落到 abort。

可说：项目同时具备事件 Runtime 和计划-执行-失败审查结构；能讨论静态规则、模型规划与恢复策略的分层。

不可说：所有任务都有持久检查点、exactly-once 事件语义，或 Manager 能自动修复全部错误。源码中的时间/随机 ID 也限制了完全确定性重放。

## 8. C 层扩展：检查点、回放、背压、并发与可观测性

### 8.1 检查点

检查点不是把整个 Python 对象 pickle 一下，而应保存版本化状态、已提交副作用、待执行动作、预算和事件 offset。恢复时要区分：

- 已计划未执行；
- 已执行但结果未持久化；
- 结果已持久化但事件未确认。

副作用工具必须用幂等键或外部事务状态消歧。

### 8.2 事件回放

理想状态由事件归约得到：

```text
state_n = reduce(apply_event, initial_state, events[0:n])
```

要实现稳定回放，事件 ID、时间源、序列、输入和归约函数必须确定；外部工具结果应记录引用或快照。随机 ID 和 `Date.now()` 需要在测试中注入。

### 8.3 背压

当生产事件速度高于消费速度，不能无限加队列。策略包括：队列上限、按租户配额、低优先级丢弃、合并流式 token 事件、暂停 Provider、拒绝新任务。背压是一致性和容量问题，不只是性能优化。

### 8.4 并发调度

计划 DAG 可按入度为 0 的任务并行。并行收益上限受最慢关键路径控制；共享工具限流、Provider 并发和上下文合并可能让理论并行失效。要记录每个节点 start/end、依赖和重试，而不是只看总耗时。

### 8.5 可观测指标

- 完成率、终止原因分布、平均/尾部 step 数；
- Provider 与 Tool P50/P95/P99 延迟；
- retry、fallback、no-progress、permission-denied 比例；
- Token/费用/任务；
- 队列深度、等待时间、并发槽位；
- 轨迹重放一致率和人工接管率。

## 9. 示例代码与故障推演（非运行实验）

> [!warning] 教学示例
> 以下代码展示受控 Loop 的责任划分，未在本课程运行。真实实现还需异步取消、持久检查点、并发安全、指标与安全测试。

```python
from dataclasses import dataclass
from time import monotonic

@dataclass(frozen=True)
class Limits:
    max_steps: int = 8
    deadline_seconds: float = 30.0
    max_no_progress: int = 2

def run(goal, provider, router, tools, policy, limits: Limits):
    state = new_state(goal)
    deadline = monotonic() + limits.deadline_seconds

    while state.step < limits.max_steps:
        if monotonic() >= deadline:
            return terminate(state, "deadline_exceeded")

        response = provider.complete(build_context(state), tools.schemas())
        decision = router.decide(response, state)

        if decision.kind == "final":
            return verify_and_finish(state, decision.text)
        if decision.kind == "abort":
            return terminate(state, decision.reason)

        for call in decision.tool_calls:
            auth = policy.authorize(call, state)
            result = tools.execute(call, auth)
            state = append_result(state, call.id, result)

        state = advance_and_check_progress(state)
        if state.no_progress_count >= limits.max_no_progress:
            return terminate(state, "no_progress")

    return terminate(state, "max_steps")
```

审查点：deadline 使用单调时钟；权限在执行前；ToolResult 关联 call.id；步数由 Runtime 增加；无进展独立于 max_steps。缺口：Provider 异常分类、并行、人工暂停和副作用幂等未展开。

## 10. 文字 Demo：三条轨迹和一个失败反例

### 10.1 ESGQA 图轨迹

**Demo ID**：`DEMO-ESG-01`。

```text
START
-> memory
-> intent(task_type=calculation)
-> retriever
-> fetcher
-> compliance
-> calculator
-> generator
-> evaluator
-> export
-> END
```

若 retriever 写入 `error_msg`，`check_retriever` 直接选择 fallback。控制流由图决定，模型不能绕过 compliance。

### 10.2 agentHub busy 轨迹

**Demo ID**：`DEMO-HUB-01`。

```text
prompt A -> Provider busy
prompt B -> queue
A done -> processNextOrIdle
检查 Provider 是否变化
发送 B
B done -> queue empty
启动 idle timer -> stop/detach
```

关键点：同一有状态 Provider 不被两个 prompt 随意并发；配置变化在边界切换。

### 10.3 Planner 失败轨迹

**Demo ID**：`DEMO-HUB-08`。

Planner 输出非 JSON -> 校验失败 -> 有限重试 -> 仍失败则抛错。执行任务失败后 Manager 输出非法 action -> Runtime 将其降级为 abort。这里的“保守失败”优于无限重规划。

### 10.4 重复副作用反例

模型两次调用 `create_order(order_id=42)`；第一次网络超时但订单实际已创建。若 Runtime 直接重试，会产生重复订单。正确设计：使用业务幂等键 42，超时后先查询订单状态，再决定是否补偿或重试。

**一分钟讲解稿**：

“我把 Agent Loop 当状态机，不当 Prompt 技巧。while-loop 适合短闭环，StateGraph 适合可枚举业务路径，事件 Runtime 适合长生命周期和外部进程。策略层可以是 ReAct、Plan-Execute 或 ReWOO，但运行时始终负责权限、预算、重试和终止。ESGQA 是显式图，agentHub 是事件 Runtime 加 Planner/Manager；二者说明控制流模式要根据任务可预测性和生命周期选择。”

## 12. 30 秒、2 分钟与深挖回答

### 12.1 30 秒版本

“Agent Loop 是 Runtime 掌控的状态转移协议。while-loop 简单，StateGraph 显式，事件 Runtime 适合长生命周期。ReAct 观察后逐步修正，Plan-Execute 先拆任务，ReWOO 把计划与观察解耦。无论用哪种策略，Runtime 都要控制 max_steps、deadline、预算、no-progress、权限和结构化终止。”

### 12.2 2 分钟版本

先画 `state -> context -> provider -> route -> tool -> observation -> state`。解释五类停止条件为何独立；再比较三种控制流和三种策略。项目落点：ESGQA 用 StateGraph 固定合规主链和 fallback；agentHub Runtime 管 Provider 队列、事件和 idle，Planner/Manager 处理计划校验和失败决策。最后讲副作用工具的幂等和检查点恢复。

### 12.3 深挖评分点

- 区分控制流载体与推理策略。
- 能写出终止不变量和失败分类。
- 提到单调时钟、幂等键、call_id、事件因果关系。
- 能从两个项目源码定位证据。
- 不把论文的基准收益外推到项目。

危险回答：

- “ReAct 就是 while True。”
- “失败就重试三次。”
- “StateGraph 天然不会死循环。”
- “用了消息队列就 exactly-once。”
- “模型会根据上下文自己停下来。”

## 13. 课后复习：口述提纲、闪卡、自测与易错点

### 13.1 口述提纲

1. 三种控制流载体。
2. ReAct/Plan-Execute/ReWOO。
3. 五类停止条件。
4. 重试与副作用。
5. 检查点、事件回放与背压。
6. ESGQA/agentHub 对照。

### 13.2 闪卡

- Q：max_steps 与 deadline？ A：前者限制离散轮数，后者限制真实时间。
- Q：no-progress 的输入？ A：规范化动作、参数、结果状态和关键状态版本。
- Q：ReWOO 的核心？ A：Planner 先产含变量的计划，Worker 执行，Solver 汇总，减少观察间反复推理。
- Q：事件回放需要什么？ A：稳定事件 ID/序列、确定性归约、外部结果快照或引用。
- Q：写工具超时为何危险？ A：副作用可能已经发生。
- Q：agentHub Runtime 不是 while-loop 的证据？ A：Provider 事件回调、队列和 idle 生命周期驱动。

### 13.3 自测

1. 为一个研究 Agent 定义步骤、时间、Token、工具和无进展预算。
2. 设计一次“执行成功但回执丢失”的恢复流程。
3. 把 ESGQA 中 calculation 分支改画成子图，说明检查点位置。
4. 为 agentHub 事件增加确定性重放所需字段。
5. 给出 ReAct 不如固定 Workflow 的两个业务例子。

### 13.4 易错点

- 把控制流框架与策略范式混为一谈。
- 重试不区分瞬时/永久/副作用。
- 只设 max_steps，不设 deadline 和 budget。
- 事件有日志但无因果 ID。
- 计划并行忽略数据依赖和限流。
